JSON-RPC vs REST in 2026: Why the Old Protocol Is Winning AI-Native APIs

Published: 2026-04-24 · 6-7 min read

JSON-RPC was declared obsolete multiple times over the past decade. REST won the enterprise API wars. GraphQL promised to end the debate. Yet in 2026, JSON-RPC is experiencing a remarkable renaissance — not despite modern AI tooling, but because of it. The Model Context Protocol (MCP), Language Server Protocol (LSP), and most major AI agent frameworks all chose JSON-RPC. This is not a coincidence.

The Core Difference

REST models APIs as resources with HTTP verbs. JSON-RPC models APIs as function calls. This distinction turns out to matter enormously when building AI agents that need to invoke tools:

AspectRESTJSON-RPC
Mental modelResources & stateProcedure calls
URL structureNouns (/users/42/posts)Single endpoint (/api)
Request formatHTTP verb + URL + bodyJSON with method + params
Batch requestsMultiple HTTP requestsNative JSON array
Bi-directionalNo (request-response only)Yes (over WebSocket)
Tool discoveryOpenAPI specBuilt into protocol

Why AI Agents Prefer JSON-RPC

When an AI agent needs to call a tool, it needs to answer three questions: what tools exist, what parameters they take, and what they return. JSON-RPC answers all three natively through its listTools and callTool patterns in MCP:

// JSON-RPC request: list available tools
{
    "jsonrpc": "2.0",
    "id": 1,
    "method": "tools/list"
}

// Response: self-describing tool catalog
{
    "jsonrpc": "2.0",
    "id": 1,
    "result": {
        "tools": [
            {
                "name": "read_file",
                "description": "Read the contents of a file at a given path",
                "inputSchema": {
                    "type": "object",
                    "properties": {
                        "path": {"type": "string", "description": "File path to read"}
                    },
                    "required": ["path"]
                }
            }
        ]
    }
}

With REST, you'd need a separate OpenAPI discovery endpoint, an HTTP client library, verb-based routing logic, and a response mapping layer. JSON-RPC collapses all of this into a single JSON structure that an LLM can reason about directly.

Batch Processing: JSON-RPC's Killer Feature

JSON-RPC's native batch support is another major advantage for AI pipelines. When an agent needs to call five tools in parallel, REST requires five HTTP round-trips with connection overhead. JSON-RPC sends one request:

// JSON-RPC batch: 5 tool calls in one request
[
    {"jsonrpc": "2.0", "id": 1, "method": "read_file",    "params": {"path": "/src/index.ts"}},
    {"jsonrpc": "2.0", "id": 2, "method": "list_dir",     "params": {"path": "/src"}},
    {"jsonrpc": "2.0", "id": 3, "method": "search_code",  "params": {"query": "fetchUser"}},
    {"jsonrpc": "2.0", "id": 4, "method": "get_git_log",  "params": {"limit": 10}},
    {"jsonrpc": "2.0", "id": 5, "method": "run_tests",    "params": {"filter": "auth"}}
]

// All results return in one response array
// Server can execute in parallel, return in any order (matched by id)

When REST Still Wins

JSON-RPC is not the right choice for every API. REST continues to dominate in several scenarios:

  • Public-facing web APIs: RESTful URLs are more human-readable and cache-friendly with HTTP infrastructure
  • Resource-centric data: When your API is truly about CRUD operations on resources, REST's semantics are cleaner
  • Browser-native requests: REST maps naturally to fetch() and existing HTTP tooling
  • CDN caching: GET endpoints in REST can be edge-cached; JSON-RPC POST-only cannot
  • Team familiarity: Most developers know REST; JSON-RPC requires onboarding

The Nordic APIs Survey Says

Nordic APIs published their April 2026 developer survey on JSON-RPC vs REST adoption patterns. Among teams building AI-integrated backends:

  • 67% are adding JSON-RPC endpoints alongside existing REST APIs
  • 31% are building new AI tool servers exclusively in JSON-RPC (via MCP)
  • Only 2% are migrating existing REST APIs to JSON-RPC (migration cost too high)

The pragmatic conclusion: most teams are running both, using REST for public-facing APIs and JSON-RPC for AI agent tool interfaces.

Practical Guide: Choosing in 2026

Choose JSON-RPC when:

  • Building MCP-compatible tools for AI agents
  • Implementing a Language Server Protocol (LSP) extension
  • Designing internal service-to-service APIs with complex method signatures
  • Need bidirectional communication over WebSocket
  • Batch operations are a primary use case

Choose REST when:

  • Building a public API that third-party developers will consume
  • Cacheability and CDN optimization matter
  • Your team has deep REST expertise and the API is resource-oriented
  • You need standard HTTP tooling (OpenAPI, Swagger UI, Postman)

Key Takeaways

  • JSON-RPC is the default protocol for AI agent tools in 2026 — MCP's success proves this
  • The protocol's self-describing nature makes it ideal for LLM tool discovery without extra documentation
  • Batch request support gives JSON-RPC a performance edge for multi-tool AI pipelines
  • REST isn't going away — it's complementary for public-facing, cacheable, resource-oriented APIs
