OpenAPI 4.0 Adopts Full JSON Schema 2020-12: Shared Schemas for API Specs and Runtime Validation

Home / News / OpenAPI 4.0 JSON Schema
Published: April 27, 2026 · 6-7 min read

The OpenAPI Initiative has released OpenAPI 4.0 with a fundamental architectural change: the specification now fully adopts JSON Schema Draft 2020-12 as its schema vocabulary. This eliminates the long-standing divergence between OpenAPI's "subset of JSON Schema" and the real JSON Schema specification — enabling shared schemas between API documentation and runtime validation for the first time.

The Problem: OpenAPI 3.x's Broken Schema Promise

Since OpenAPI 3.0, the specification has claimed to use "a subset of JSON Schema Draft 04" — but in practice, the subset was so heavily modified that OpenAPI schemas were incompatible with standard JSON Schema validators. Key incompatibilities included:

  • nullable vs null type — OpenAPI used nullable: true while JSON Schema uses { "type": ["string", "null"] }
  • discriminator not in JSON Schema — OpenAPI's polymorphism mechanism had no JSON Schema equivalent
  • readOnly / writeOnly — OpenAPI extensions not recognized by JSON Schema validators
  • format behavior — OpenAPI required format validation; JSON Schema made it optional
  • No $dynamicRef — OpenAPI couldn't use JSON Schema's most powerful composition feature

The result: teams maintaining OpenAPI specs had to maintain two separate schemas — one for documentation (OpenAPI) and one for runtime validation (JSON Schema). This duplication was error-prone and defeated the purpose of a single source of truth.

OpenAPI 4.0: Full JSON Schema Compatibility

OpenAPI 4.0 resolves this by adopting the complete JSON Schema Draft 2020-12 vocabulary. An OpenAPI 4.0 schema is now a valid JSON Schema that can be used directly with any compliant validator — Ajv, json-schema-validator, or the built-in validators in Deno, Bun, and Cloudflare Workers.

Here's what the new schema syntax looks like:

{
  "openapi": "4.0.0",
  "info": { "title": "User API", "version": "1.0.0" },
  "paths": {
    "/users": {
      "post": {
        "requestBody": {
          "content": {
            "application/json": {
              "schema": {
                "$ref": "#/$defs/UserCreate"
              }
            }
          }
        },
        "responses": {
          "201": {
            "content": {
              "application/json": {
                "schema": {
                  "$ref": "#/$defs/User"
                }
              }
            }
          }
        }
      }
    }
  },
  "$defs": {
    "UserCreate": {
      "type": "object",
      "properties": {
        "name": { "type": "string", "minLength": 1 },
        "email": { "type": "string", "format": "email" },
        "role": { "$ref": "#/$defs/Role" }
      },
      "required": ["name", "email"],
      "additionalProperties": false
    },
    "User": {
      "$dynamicRef": "#/$defs/UserCreate",
      "properties": {
        "id": { "type": "string", "format": "uuid" },
        "createdAt": { "type": "string", "format": "date-time" }
      }
    },
    "Role": {
      "type": "string",
      "enum": ["admin", "editor", "viewer"]
    }
  }
}

Key improvements visible in this example:

  • $defs replaces definitions — following JSON Schema 2020-12 convention
  • $dynamicRef — OpenAPI can now use dynamic references for schema composition and polymorphism, replacing the old discriminator keyword
  • additionalProperties: false — works identically in both OpenAPI and JSON Schema validators
  • No more nullable — use "type": ["string", "null"] as JSON Schema intended

The Single Schema Workflow

With OpenAPI 4.0, teams can now use the exact same schema file for:

  1. API documentation — render specs in Swagger UI, Redoc, or Elements
  2. Code generation — generate TypeScript types, Python Pydantic models, Go structs
  3. Runtime validation — validate request/response payloads with Ajv, jsonschema, or runtime-native validators
  4. Contract testing — use Pact or Schemathesis to verify API compliance
  5. AI agent integration — expose schemas as function definitions for LLM tool calling

This eliminates an entire class of bugs where the OpenAPI spec and the runtime validation schema diverged silently — a problem that affected an estimated 34% of production APIs according to a 2025 SmartBear survey.

Migration from OpenAPI 3.1

The OpenAPI Initiative provides a migration tool (openapi-migrate) that handles the most common conversion patterns:

# Install the migration tool
npm install -g @openapi-tools/migrate

# Migrate a 3.1 spec to 4.0
openapi-migrate --from 3.1 --to 4.0 ./openapi.yaml --output ./openapi-v4.yaml

# Validate the migrated spec
openapi-migrate --validate ./openapi-v4.yaml

The migration tool handles these automatic transformations:

  • nullable: true → "type": ["originalType", "null"]
  • discriminator → $dynamicRef with if/then guards
  • definitions → $defs
  • readOnly / writeOnly → x-readOnly / x-writeOnly extension annotations

Tool Ecosystem Support

Major tools in the OpenAPI ecosystem have already announced 4.0 support:

  • Swagger UI 6.0 — full rendering of OpenAPI 4.0 specs, including $dynamicRef
  • Redocly — linting, bundling, and documentation generation for 4.0
  • OpenAPI Generator 8.0 — code generation with proper $dynamicRef support
  • Stoplight Studio — visual editor with real-time JSON Schema validation
  • Ajv v9 — validates OpenAPI 4.0 schemas directly, no conversion needed

Why This Matters for AI APIs

The convergence of OpenAPI and JSON Schema has an unexpected beneficiary: AI function calling. LLM providers (OpenAI, Anthropic, Google) all use JSON Schema to define tool parameters. With OpenAPI 4.0, an API's parameter schema can be directly reused as an LLM tool definition — no manual translation required.

This means that any OpenAPI 4.0 API is automatically AI-agent-ready — a significant advantage as MCP (Model Context Protocol) and similar frameworks make JSON Schema the lingua franca of AI tool integration.
