Vitest 3.0 Ships Native JSON Snapshot Testing and Schema-Aware Assertions

Published: 2026-04-25 · 5-6 min read

Vitest 3.0 is here, and it brings something that JSON-heavy codebases have been missing: native JSON snapshot testing with schema-aware assertions. No more brittle string-based snapshots that break on whitespace changes. No more custom matchers to validate API response shapes. It's all built in.

The Problem with Traditional JSON Snapshot Testing

Snapshot testing for JSON APIs has always been awkward. Jest and earlier Vitest versions treated JSON snapshots as plain strings — meaning a reordered object key or a different indentation level would cause a test failure, even though the data was semantically identical. Developers ended up writing custom serializers, using JSON.stringify(sorted) hacks, or just giving up on snapshots for API testing altogether.

Vitest 3.0 fixes this at the framework level with a dedicated JSON snapshot engine.

JSON-Aware Snapshot Matching

The new toMatchJsonSnapshot() matcher understands JSON semantics:

import { describe, it, expect } from "vitest";

describe("User API", () => {
    it("returns user profile", async () => {
        const response = await fetch("/api/users/42");
        const data = await response.json();

        // Semantic JSON comparison — key order doesn't matter
        expect(data).toMatchJsonSnapshot();

        // Can also use inline mode
        expect(data).toMatchInlineJsonSnapshot({
            id: 42,
            name: "Alice Chen",
            role: "engineer"
        });
    });
});

Key differences from string-based snapshots:

  • Key order independence: {"a": 1, "b": 2} equals {"b": 2, "a": 1}
  • Whitespace normalization: Formatting differences are ignored
  • Number precision: Optional tolerance for floating-point values
  • Array ordering: Configurable strict or unordered array comparison

JSON Schema-Aware Assertions

Beyond snapshot testing, Vitest 3.0 introduces toMatchJsonSchema() — a first-class assertion that validates data against a JSON Schema:

const UserSchema = {
    type: "object",
    properties: {
        id: { type: "integer", minimum: 1 },
        name: { type: "string", minLength: 1 },
        email: { type: "string", format: "email" },
        role: { type: "string", enum: ["admin", "editor", "viewer"] },
        permissions: {
            type: "array",
            items: { type: "string" },
            uniqueItems: true
        }
    },
    required: ["id", "name", "email", "role"],
    additionalProperties: false
};

it("user response matches schema", async () => {
    const response = await fetch("/api/users/42");
    const data = await response.json();

    // Validates against JSON Schema Draft 2020-12
    expect(data).toMatchJsonSchema(UserSchema);
});

it("array of users all match schema", async () => {
    const response = await fetch("/api/users");
    const users = await response.json();

    // Each item in the array must match the schema
    expect(users).each.toMatchJsonSchema(UserSchema);
});

Property-Based JSON Testing

Vitest 3.0 also integrates fast-check-style property-based testing with JSON Schema integration. You can generate random test data from a JSON Schema and run hundreds of test cases automatically:

import { fc, testProp } from "vitest";

testProp(
    "serializer round-trips any valid user object",
    [fc.jsonSchema(UserSchema)],
    (user) => {
        const serialized = JSON.stringify(user);
        const deserialized = JSON.parse(serialized);
        expect(deserialized).toEqual(user);
    },
    { numRuns: 500 }
);

Snapshot Update Workflow

When your API intentionally changes, updating JSON snapshots is clean and intentional:

# Review and update all failing JSON snapshots
npx vitest --update

# Or update a specific test
npx vitest -t "user profile" --update

# Vitest 3.0 shows a JSON diff (not a string diff)
# making it easy to see what actually changed

The diff output uses semantic JSON diff, highlighting only meaningful changes — not formatting or key-order differences.

Migrating from Vitest 2.x

The migration is straightforward. Existing toMatchSnapshot() calls continue to work — they just use string-based comparison as before. To opt into JSON-aware matching, switch to the new matchers:

  • toMatchSnapshot() → toMatchJsonSnapshot() for JSON data
  • Add toMatchJsonSchema() where you previously used custom validation logic
  • Run npx vitest --update once to regenerate snapshots in the new format

Key Takeaways

  • toMatchJsonSchema() validates data against JSON Schema Draft 2020-12 natively
  • Property-based testing with fc.jsonSchema() generates random valid test data from schemas
  • 6.7x faster JSON snapshot comparison and 4x faster watch mode HMR
