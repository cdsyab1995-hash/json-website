Effect-TS 3.0: First-Class JSON Schema Generation and Runtime Validation

Published: 2026-04-28 · 6-7 min read

Effect-TS 3.0 is out, and its headline feature is a completely rebuilt Schema module with first-class JSON Schema 2020-12 generation, runtime validation with branded types, and automatic OpenAPI 3.1 spec output — all derived from a single type definition. The functional TypeScript ecosystem now has a native answer to Zod that integrates seamlessly with Effect's composition model.

Schema-First: One Definition, Three Outputs

The core principle of Effect-TS 3.0's schema system is define once, use everywhere. A single Schema definition simultaneously gives you:

  1. A TypeScript type for compile-time safety
  2. A runtime validator for parsing untrusted JSON
  3. A JSON Schema document for API documentation and tooling
import { Schema as S, JSONSchema } from "@effect/schema";

// Define once
const UserSchema = S.Struct({
  id: S.UUID,
  name: S.NonEmpty(S.String),
  email: S.pattern(S.String, /^[^\s@]+@[^\s@]+\.[^\s@]+$/),
  age: S.Int.pipe(S.between(0, 150)),
  role: S.Literal("admin", "user", "moderator"),
  createdAt: S.Date,
});

// TypeScript type — inferred automatically
type User = S.Schema.Type<typeof UserSchema>;
// { id: string; name: string; email: string; age: number; role: "admin"|"user"|"moderator"; createdAt: Date }

// Runtime parse (throws on failure)
const user = S.decodeUnknownSync(UserSchema)(rawJsonData);

// JSON Schema 2020-12 output
const jsonSchema = JSONSchema.make(UserSchema);
// { "$schema": "https://json-schema.org/draft/2020-12/schema", "type": "object", ... }

Branded Types: JSON Validation Beyond Primitives

One of Effect-TS 3.0's most powerful additions is branded type integration in the schema layer. Branded types let you distinguish semantically different strings (a UserId vs a ProductId) at the type level — and now, the schema system enforces these brands at parse time:

import { Schema as S, Brand } from "@effect/schema";

// Define branded types
const UserId = S.String.pipe(S.brand("UserId"));
const ProductId = S.String.pipe(S.brand("ProductId"), S.startsWith("prod_"));

type UserId = S.Schema.Type<typeof UserId>;    // string & Brand<"UserId">
type ProductId = S.Schema.Type<typeof ProductId>; // string & Brand<"ProductId">

const OrderSchema = S.Struct({
  orderId: S.UUID.pipe(S.brand("OrderId")),
  userId: UserId,
  productId: ProductId,  // must start with "prod_" — validated at parse time
  quantity: S.Positive(S.Int),
});

// TypeScript prevents mixing up UserId and ProductId — even though both are strings
function getUser(id: UserId) { /* ... */ }
const order = S.decodeUnknownSync(OrderSchema)(rawData);
getUser(order.data.productId); // TS error: ProductId is not assignable to UserId ✓

This eliminates an entire class of bugs where semantically different IDs are passed to the wrong function — caught at compile time, not in production.

Automatic OpenAPI 3.1 Spec Generation

Effect-TS 3.0 ships a new @effect/openapi package that generates complete OpenAPI 3.1 specifications from your schema definitions. Since OpenAPI 3.1 uses JSON Schema 2020-12 natively, the translation is lossless:

import { OpenAPI } from "@effect/openapi";
import { HttpRouter } from "@effect/platform";

const router = HttpRouter.empty.pipe(
  HttpRouter.post("/users", {
    request: { body: UserSchema },
    response: { 200: UserSchema, 422: ValidationErrorSchema },
    handler: createUserHandler,
  }),
  HttpRouter.get("/users/:id", {
    request: { params: S.Struct({ id: UserId }) },
    response: { 200: UserSchema, 404: S.Struct({ message: S.String }) },
    handler: getUserHandler,
  })
);

// Generate complete OpenAPI 3.1 spec — zero manual YAML
const spec = OpenAPI.fromRouter(router);
// Output: valid openapi.json with all schemas, paths, and response types

Teams using Effect-TS for their HTTP layer can now eliminate hand-written OpenAPI YAML entirely — the spec is always in sync with the code because it is the code.

Error Messages: Human-Readable Validation Failures

Effect-TS 3.0 completely rewrote its error formatting engine. Validation failures now produce structured, human-readable error trees rather than cryptic type errors:

const result = S.decodeUnknown(UserSchema)({
  id: "not-a-uuid",
  name: "",
  email: "not-an-email",
  age: 200,
  role: "superadmin"
});

// Effect.runSync will produce a readable ParseError:
// ParseError: 5 errors found
//   ├── id: Expected UUID, received "not-a-uuid"
//   ├── name: Expected NonEmpty string, received ""
//   ├── email: Does not match pattern /^[^\s@]+@[^\s@]+\.[^\s@]+$/
//   ├── age: Expected integer between 0 and 150, received 200
//   └── role: Expected "admin" | "user" | "moderator", received "superadmin"

The structured error format is also available as JSON for programmatic handling — critical for APIs that need to return RFC 9457 Problem Details responses with field-level validation errors.

Migration from Zod

The Effect team published a Zod compatibility layer that allows incremental migration:

  • z.string() → S.String
  • z.object({...}) → S.Struct({...})
  • z.array(z.string()) → S.Array(S.String)
  • z.union([...]) → S.Union(...)
  • z.infer<typeof schema> → S.Schema.Type<typeof schema>
  • schema.parse(data) → S.decodeUnknownSync(schema)(data)

For new projects, Effect-TS 3.0's schema module offers a compelling unified approach: one library for async workflows, dependency injection, error handling, and now JSON validation — all with world-class TypeScript inference.
