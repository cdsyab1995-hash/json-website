serde_json v2.0 Ships Zero-Copy Parsing and SIMD Acceleration: The Fastest JSON Library in Any Language

Published: 2026-04-26 · 6-7 min read

The Rust ecosystem's most widely-used JSON library, serde_json, has released version 2.0 with a set of features that fundamentally change what a JSON library can do. The headline additions — zero-copy parsing, SIMD-accelerated tokenization, and a new streaming JSON API — push serde_json well beyond any previous performance ceiling and make it competitive with hand-optimized C implementations.

Zero-Copy Parsing: Memory Without Allocations

The traditional serde_json parsing flow allocates a new tree of owned Value structs for every document — meaning heap allocations proportional to the number of keys and values in the JSON. For high-throughput services processing millions of small JSON documents per minute, this creates significant GC pressure and memory churn.

serde_json v2.0 introduces zero-copy parsing via the new serde_json::from_slice_zero_copy function. Instead of owning the parsed data, the resulting Value borrows directly from the input bytes — zero allocations for the parse itself, and zero copies of string data:

use serde_json::{self, Value};

let json_bytes = std::fs::read("data.json")?;
// Zero allocation parse — Value borrows from json_bytes
let value: Value = serde_json::from_slice_zero_copy(&json_bytes)?;

println!("{}", value["name"]); // Still valid while json_bytes is alive

This works by storing a reference to the original byte buffer alongside the Value tree. As long as the source buffer is alive, all string slices in the parsed tree remain valid — no String allocations, no lifetime complexity for the caller.

SIMD Tokenization: 4x Faster Parsing

serde_json v2.0's tokenizer is now SIMD-accelerated using portable SIMD intrinsics (std::simd). On x86 and ARM64 targets, the tokenizer processes 32 bytes per cycle using AVX-256 or NEON instructions — compared to the single-byte-at-a-time approach of version 1.x.

Benchmarks published by the serde_json team, comparing v1.0.143 to v2.0.0:

100 bytes (API request)0.8µs0.7µs1.1x
1KB (typical API response)5.2µs1.8µs2.9x

For typical API workloads (1-10KB payloads), parsing is 3-4x faster. For large file processing, the improvement reaches 5x. This makes serde_json v2.0 faster than many C-based JSON libraries, while retaining Rust's safety guarantees.

New Streaming API: Processing Gigabyte-Scale JSON

serde_json v2.0 ships a completely new streaming parser designed for large documents that don't fit in memory. The new JsonStream API yields parsed JSON fragments incrementally — allowing processing of gigabyte-scale JSON files with constant memory usage:

use serde_json::stream::{JsonStream, Event};

let file = std::fs::File::open("huge.json")?;
let reader = std::io::BufReader::new(file);
let mut stream = JsonStream::from_reader(reader);

while let Some(event) = stream.next() {
    match event? {
        Event::StringValue(s) => println!("String: {}", s),
        Event::Number(n) => println!("Number: {}", n),
        Event::ObjectStart => println!("Object started"),
        Event::ObjectEnd => println!("Object ended"),
        _ => {}
    }
}

Unlike the previous StreamDeserializer, the new JsonStream doesn't require specifying a target struct type upfront — it yields typed events directly, making it suitable for JSON transformation, filtering, and re-serialization pipelines.

API Stability and Migration

serde_json v2.0 is a new major version with breaking changes. Key migration points:

  • from_str, from_slice, to_string — remain unchanged and are still the recommended API for most use cases
  • from_slice_zero_copy — new, opt-in zero-copy mode. Not enabled by default to preserve backwards compatibility
  • stream::Deserializer — replaced by stream::JsonStream
  • The value module is now a re-export of the serde_json::Value type alias
  • Requires Rust 1.85+ (for std::simd and async trait improvements)

Ecosystem Impact

serde_json v2.0 is expected to have broad ecosystem impact. As the foundation of most Rust JSON processing, the performance improvements cascade through:

  • axum and actix-web — faster JSON request/response handling
  • reqwest — faster JSON deserialization in HTTP clients
  • tokio-based data pipelines — lower latency for JSON streaming workloads
  • Data ingestion tools — 5x faster bulk JSON processing

The crate is available on crates.io as serde_json = "2.0". The migration guide is at serde.rs/json/v2.
