JSON Schema 2026 Update: Draft 2020-12 Becomes the Unchallenged Standard

Published: 2026-05-06 · 4 min read

The JSON Schema specification took a major leap forward when IETF published RFC 9629 in early 2026, which normatively references JSON Schema Draft 2020-12 and elevates it from a community specification to an officially recognized standard. This publication removed the last organizational hesitation about adopting JSON Schema in enterprise environments - legal, procurement, and standards-compliance teams that previously required IETF or W3C backing now have clear justification.\n\nMajor cloud providers have responded by making Draft 2020-12 the default in their managed services. AWS API Gateway now validates incoming requests against JSON Schema Draft 2020-12 by default. Google Cloud Endpoints and Azure API Management have similarly updated their validators. This default shift means that new JSON Schema documents authored today should assume Draft 2020-12 features are universally available.

dynamicRef and Anchor Features in Production

The dynamicRef feature, which allows schemas to reference other schemas using runtime evaluation rather than static resolution, has become the most adopted Draft 2020-12 feature in 2026. Real-world usage centers on polymorphic schemas where the validation rules depend on a discriminator field. A payment processing API using dynamicRef can define a base transaction schema and specialized schemas for credit card, bank transfer, and cryptocurrency transactions, with the correct schema automatically selected based on a type discriminator.\n\nJSON Schema anchors (using $anchor) provide named sub-schema shortcuts within a document, complementing dynamicRef by allowing frequently-referenced patterns to be given memorable names. The anchor system also supports recursive schemas through the new $recursiveAnchor keyword.

unevaluatedProperties: The Validation Improvement Everyone Wanted

The unevaluatedProperties keyword has replaced additionalProperties as the recommended way to reject unexpected fields in validated objects. Unlike additionalProperties (which ignores the effect of $ref, if/then/else, and other keywords when determining which properties are allowed), unevaluatedProperties accurately tracks which properties have been explicitly validated by any keyword in the schema.\n\nThis distinction matters enormously for API schema evolution. With additionalProperties: false, extended fields would be incorrectly rejected. With unevaluatedProperties: false, those fields are correctly permitted. All major JSON Schema validators have now optimized unevaluatedProperties performance to match additionalProperties.

contentSchema and Content Encoding Transform JSON

Draft 2020-12s contentSchema feature, combined with base64 content encoding, enables JSON Schema to describe binary data embedded in JSON documents. A schema can now specify that a field named thumbnail contains a PNG image, encoded as base64, with a minimum size of 1KB and maximum of 10MB. Validators can optionally decode and validate the binary content against the declared media type.\n\nThe contentEncoding (base64) and contentMediaType (image/png) keywords work together to declaratively specify encoding. In 2026, these features are most commonly used in webhook payload schemas, where binary signatures or encrypted tokens are embedded in JSON.

Community Resources and Learning Path

The JSON Schema community has invested heavily in documentation and learning resources in 2026. The official JSON Schema website now hosts an interactive tutorial that walks through Draft 2020-12 features with live examples. The Understanding JSON Schema book has been fully updated for Draft 2020-12 and is available for free online. The JSON Schema Slack workspace (with over 40,000 members) provides fast answers to questions from schema authors at all experience levels.\n\nFor teams adopting JSON Schema for the first time, the recommended learning path starts with basic type validation, progresses to object and array constraints, then covers string formats and pattern matching, before tackling advanced topics like $ref, conditional logic (if/then/else), and dynamic references.

Migration from Draft-07: What Changed

Organizations still running JSON Schema Draft-07 are finding 2026 the right time to migrate. The key breaking changes are limited: $id vs id (now consistently $id), the location of definitions ($defs replacing definitions inside Draft 2020-12), and uri-template relative references. The JSON Schema team provides an official migration guide with an automated compatibility checker.\n\nMost Draft-07 schemas migrate with minimal changes. The most common migration issue is $schema URIs: changing from http://json-schema.org/draft-07/schema# to https://json-schema.org/draft/2020-12/schema. OpenAPI 3.1 already requires Draft 2020-12, making it a forcing function for API-first organizations.

Validation Performance: Benchmarks in 2026

JSON Schema validator performance has improved significantly in 2026, driven by optimizations for Draft 2020-12 features and competition among validators. Ajv (the dominant JavaScript validator) now processes complex schemas with dynamicRef at speeds comparable to Draft-07 equivalent schemas.\n\nThe Rust-based jsonschema validator has emerged as the performance leader for CLI tools and server-side applications, outperforming Python jsonschema by 10-50x on large test suites. Go gojsonschema and the native-json library for Java have similarly improved.

Looking Ahead: JSON Schema 2028 and Beyond

The JSON Schema working group has begun preliminary discussions for the post-Draft 2020-12 specification cycle, tentatively targeting a 2028 release. Early proposals include a declarative constraint language that could replace if/then/else chains with a more readable syntax, improved support for tuple validation in arrays, and a standard extension mechanism for custom keywords.\n\nThe most anticipated proposal is improved support for embedded schemas and code generation hints. The contentSchema mechanism has shown that JSON Schema can describe not just data structure but also data meaning, and the next iteration may formalize this into a richer annotation system.
