JSON API Security in 2026: 10 Practices You Can't Afford to Ignore

Published: 2026-04-29 · 8-10 min read

The OWASP API Security Top 10 lists broken object-level authorization and excessive data exposure as the #1 and #2 most critical API vulnerabilities. Both directly relate to how your API handles JSON data. These aren't theoretical concerns — they're the attack vectors behind the breaches you read about. Here's how to protect your JSON APIs in 2026.

1. 1. Validate Input at the Gate

Never trust incoming JSON data. Validate before processing:

// ❌ Dangerous: Trusting client input


app.post('/api/users', (req, res) => {


  db.query(INSERT INTO users VALUES (${req.body.id}, '${req.body.name}'));


});


// ✅ Safe: Schema validation first


app.post('/api/users', validate(schema), (req, res) => {


  db.query('INSERT INTO users VALUES (?, ?)', [req.body.id, req.body.name]);


});

Use JSON Schema with Ajv, Zod, or Pydantic to validate structure AND types.

2. 2. Prevent JSON Injection

Unsanitized JSON can inject into logs, databases, or other JSON consumers:

// ❌ Vulnerable to log injection


const user = JSON.parse(req.body.data);


logger.info(User action: ${user.name});


// ✅ Sanitize before logging


const sanitized = JSON.parse(req.body.data);


logger.info({


  message: 'User action',


  userName: sanitized.name.replace(/[\r\n]/g, ''),


  userId: sanitized.id


});


// ✅ For database: Use parameterized queries


const user = validate(userSchema, req.body);


await db.query(


  'INSERT INTO logs (type, data) VALUES (?, ?)',


  ['user_action', JSON.stringify(sanitizeForLogging(user))]


);

3. 3. Implement Rate Limiting with JSON Responses

Return structured error responses when limits are exceeded:

{


  "error": {


    "code": "RATE_LIMIT_EXCEEDED",


    "message": "Too many requests. Please wait before retrying.",


    "details": {


      "limit": 100,


      "window": "1 minute",


      "retryAfter": 45


    },


    "docs": "https://api.example.com/docs/rate-limiting"


  }


}

Use Redis for distributed rate limiting:

async function checkRateLimit(userId: string): Promise {


  const key = ratelimit:${userId};


  const current = await redis.incr(key);


  if (current === 1) {


    await redis.expire(key, 60); // 1-minute window


  }


  if (current > 100) {


    const ttl = await redis.ttl(key);


    throw new RateLimitError({


      code: 'RATE_LIMIT_EXCEEDED',


      retryAfter: ttl


    });


  }


  return true;


}

Common Mistakes to Avoid

❌ Never expose internals in errors:

{


  "error": "Database error: Connection to postgres://user:password@host/db failed"


}

✅ Return generic messages to clients:

{


  "error": {


    "code": "INTERNAL_ERROR",


    "message": "An unexpected error occurred. Reference: ABC123"


  }


}

❌ Accepting any content type:

app.use((req, res, next) => {


  // Attacker sends: Content-Type: application/x-www-form-urlencoded


  // with body: _method=DELETE


  req.body = parseBody(req);


  next();


});

✅ Strict content-type enforcement:

app.use((req, res, next) => {


  if (req.method === 'POST' || req.method === 'PUT') {


    const contentType = req.get('Content-Type');


    if (!contentType?.includes('application/json')) {


      return res.status(415).json({


        error: { code: 'UNSUPPORTED_MEDIA_TYPE', message: 'Content-Type must be application/json' }


      });


    }


  }


  next();


});

❌ No limits on JSON payload size:

app.use(express.json()); // Unlimited by default!

✅ Configure reasonable limits:

app.use(express.json({


  limit: '100kb',        // Max payload size


  strict: true,          // Reject malformed JSON


  verify: (req, res, buf) => {


    // Custom verification


    if (buf.length > 100 * 1024) {


      throw new Error('Payload too large');


    }


  }


}));

Your Implementation Checklist

## Security Checklist Before deploying any JSON API: - [ ] **Input Validation**: Schema validation on all endpoints - [ ] **Parameterized Queries**: No string concatenation in DB queries - [ ] **Rate Limiting**: Per-user and per-IP limits configured - [ ] **Content-Type Enforcement**: Strict JSON validation - [ ] **Payload Limits**: Max size configured (recommend 100KB-1MB) - [ ] **Error Sanitization**: No internals exposed in responses - [ ] **Authentication**: JWT/OAuth2 with proper validation - [ ] **Authorization**: Object-level access checks - [ ] **Logging**: Audit log for sensitive operations - [ ] **HTTPS**: TLS 1.3 enforced - [ ] **CORS**: Proper origin restrictions - [ ] **Security Headers**: CSP, X-Frame-Options, X-Content-Type-Options Validate your JSON payloads before storing.
