Modern JSON Error Handling: Patterns That Work

Published: 2026-05-05 · 2 min read · 318 words

Why JSON Error Design Matters

Error responses are where many APIs fail users most severely. When something goes wrong, developers debugging integrations need clear, actionable information: what went wrong, where it went wrong, and ideally how to fix it. Poor error handling creates frustrated developers, long debugging sessions, and support burden. Good error design reduces all three. The challenge is balancing detail (helps debugging) with security and simplicity.

RFC 7807: Problem Details for HTTP APIs

RFC 7807 defines a standard format for HTTP API error responses with type (URI linking to documentation), title (human-readable summary), detail (specific context), and instance (identifying the request) properties. Custom properties can provide additional context. This format is gaining adoption because it provides structure without being overly prescriptive. The type URI links to documentation, making errors self-documenting.

JSON:API Error Format

JSON:API defines its own error format optimized for its resource-oriented model. Errors include id (unique error identifier), links (about and describedBy URLs), status (HTTP status code), code (application-specific error code), title (short summary), detail (human-readable explanation), and source (pointer to the specific field or parameter). Relationships to related errors can be expressed. The format integrates naturally with JSON:API overall structure.

Custom Error Formats: When and Why

Many APIs use custom error formats that do not conform to RFC 7807 or JSON:API. This is not always wrong—sometimes you have specific requirements that standards do not address. Custom formats can be simpler for simple APIs. However, when building custom formats, include: a human-readable message, an error code that code can switch on, the field that caused the error (for validation), and links to documentation. Avoid exposing internal implementation details.

Error Handling Best Practices

Use appropriate HTTP status codes consistently. 400 for client errors, 401/403 for auth, 404 for missing resources, 422 for validation errors, 429 for rate limiting, 500 for server errors. Return errors in the same format as successful responses (consistency helps client code). Log errors server-side with full details for debugging; return sanitized details to clients. Include a request identifier in error responses that maps to server-side logs.
