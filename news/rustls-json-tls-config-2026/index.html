Rustls 0.23 Introduces JSON-Based TLS Configuration: Goodbye OpenSSL Config Files

Published: 2026-05-06 · 6-7 min read

TLS configuration has been one of the last holdouts of complex legacy configuration formats. Rustls 0.23 changes this by introducing a fully JSON-based TLS configuration system that supports schema validation, live reload without service restart, and programmatic certificate management.

JSON TLS Configuration in Practice

Replace complex OpenSSL configuration directives with readable JSON: 

// rustls.json - New format
{
  "server": {
    "certificates": [
      {
        "cert": "/etc/ssl/certs/api.example.com.pem",
        "key": "/etc/ssl/private/api.example.com.key",
        "domains": ["api.example.com", "*.api.example.com"]
      }
    ],
    "protocols": ["TLSv1.3", "TLSv1.2"],
    "cipherSuites": [
      "TLS_AES_256_GCM_SHA384",
      "TLS_CHACHA20_POLY1305_SHA256"
    ],
    "clientAuth": {
      "required": false,
      "caBundle": "/etc/ssl/certs/ca-bundle.pem"
    }
  }
}

Live Reload Without Downtime

The killer feature: Rustls 0.23 watches the JSON config file and hot-reloads TLS certificates on change. Certificate rotation — previously requiring service restarts or complex reload scripts — becomes trivial. New connections use the updated certificate immediately; existing connections complete with the old certificate.

Schema Validation Catches Misconfigurations

All JSON configurations are validated against a published JSON Schema before being applied. Invalid cipher suites, malformed certificate paths, and conflicting protocol settings are caught at load time with clear error messages, not discovered during TLS handshake failures in production.

Ecosystem Impact

Nginx plans JSON TLS configuration support in v1.28 (May 2026). HAProxy has implemented a compatible JSON config format. Caddy's JSON configuration already supports similar patterns. The trend toward JSON for infrastructure configuration — already dominant in Kubernetes and Terraform — is now reaching TLS.
