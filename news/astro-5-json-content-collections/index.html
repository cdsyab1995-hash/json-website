Astro 5.0 Ships Native JSON Content Collections with Auto TypeScript Types

Published: 2026-04-28 · 5-6 min read

Astro 5.0 has landed with a major upgrade to its Content Collections system — now fully supporting JSON and YAML data files as first-class collection sources with automatic TypeScript type generation, Zod schema validation, and a new Content Layer API that makes JSON-driven sites dramatically simpler to build.

JSON Collections: Zero-Config Data Management

Before Astro 5.0, content collections were primarily designed for Markdown and MDX. Developers working with JSON data had to resort to manual imports, TypeScript casting, or custom loaders. Version 5.0 eliminates this entirely — any .json file placed under src/content/ is automatically indexed and typed:

// src/content/config.ts
import { defineCollection, z } from "astro:content";

export const collections = {
  products: defineCollection({
    type: "data",   // ← new in Astro 5.0
    schema: z.object({
      id: z.string(),
      name: z.string(),
      price: z.number().positive(),
      tags: z.array(z.string()).optional(),
      inStock: z.boolean().default(true),
    }),
  }),
};

Drop a products.json (or a folder of *.json files) under src/content/products/, and Astro automatically generates typed accessors — no manual schema definitions required beyond the Zod config above.

Automatic TypeScript Type Inference

The biggest ergonomic win is Astro 5.0's new type inference engine. When you call getCollection(), the return type is automatically derived from your Zod schema — no manual casting, no as ProductType[] assertions:

// src/pages/products/[slug].astro
---
import { getCollection, getEntry } from "astro:content";

// Fully typed — inferred from your Zod schema
const products = await getCollection("products");
// Type: CollectionEntry<"products">[]
// Each entry.data is typed as:
// { id: string; name: string; price: number; tags?: string[]; inStock: boolean }

const product = await getEntry("products", Astro.params.slug);
// TypeScript error if slug is not a valid product ID ✓
---


{product.data.name}


${product.data.price.toFixed(2)}

This tight integration between Zod schemas and TypeScript types means JSON validation errors surface at build time, not at runtime in production. Invalid JSON data that violates the schema causes the build to fail with a descriptive error message — a major DX improvement for content-heavy sites.

Content Layer API: Fetch JSON from Any Source

Astro 5.0's new Content Layer API extends collections beyond local files. You can now define remote JSON data sources — REST APIs, CMS APIs, databases — that are fetched at build time and treated identically to local JSON collections:

// src/content/config.ts
import { defineCollection, z } from "astro:content";

const githubStars = defineCollection({
  loader: async () => {
    const res = await fetch(
      "https://api.github.com/repos/withastro/astro/stargazers?per_page=100"
    );
    const data = await res.json();
    // Returns JSON array — Astro indexes it automatically
    return data.map(user => ({ id: user.login, ...user }));
  },
  schema: z.object({
    id: z.string(),
    login: z.string(),
    avatar_url: z.string().url(),
    html_url: z.string().url(),
  }),
});

export const collections = { githubStars };

The loader runs once at build time (or during Astro's new incremental rebuild mode), caches the result, and makes all entries available through the same typed getCollection() API. No more fetch() calls scattered across components — JSON data is centralized, validated, and typed.

Performance: Build-Time JSON Indexing

Astro 5.0 rewrote its content indexing pipeline in Rust, and the performance gains are significant for large JSON datasets:

ScenarioAstro 4.xAstro 5.0Improvement
1,000 JSON entries2.1s0.3s7x faster
10,000 JSON entries18.4s1.1s16x faster
100,000 JSON entriesOOM crash6.8sStable
Schema validation (10k)4.2s0.4s10x faster

The Rust indexer processes JSON files in parallel, validates against Zod schemas via a compiled Wasm module, and outputs a binary index that subsequent builds can hot-reload. Sites with large product catalogs or documentation datasets see the most dramatic improvements.

JSON Query Helpers

New in Astro 5.0 are built-in query helpers that allow filtering, sorting, and paginating JSON collections without writing boilerplate array operations:

import { getCollection } from "astro:content";

// Filter by field value
const inStockProducts = await getCollection("products", 
  ({ data }) => data.inStock === true
);

// Sort by field
const byPrice = inStockProducts.sort(
  (a, b) => a.data.price - b.data.price
);

// Paginate
import { paginate } from "astro:content";
const page = paginate(byPrice, { pageSize: 12 });

Migration from Astro 4.x

For teams upgrading from Astro 4.x, the migration path is straightforward. Existing Markdown collections are unchanged. To migrate a JSON data import to the new collections API:

  • Move JSON files from src/data/ to src/content/<collection-name>/
  • Add a collection definition in src/content/config.ts with a Zod schema
  • Replace import data from "../data/items.json" with await getCollection("items")
  • Remove manual TypeScript type assertions — types are now inferred automatically

The Astro team has published an automated codemod that handles most of these changes: npx astro@5 migrate will scan your project and suggest or apply the necessary changes.

Why This Matters

Astro 5.0's JSON collections represent a fundamental shift in how static sites manage structured data. The combination of local and remote JSON sources, Zod schema validation at build time, automatic TypeScript types, and the new Rust-powered indexer makes Astro a genuinely compelling choice for JSON-heavy sites — product catalogs, documentation portals, data journalism sites — where data quality and build performance matter.
