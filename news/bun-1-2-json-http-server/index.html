Bun 1.2 Redesigns HTTP Server with Native JSON Helpers and 2x Throughput

Published: 2026-04-27 · 5-6 min read

Bun 1.2 has landed with a sweeping overhaul of its HTTP server API, introducing native JSON response helpers, automatic Content-Type negotiation, and streaming JSON support — all backed by a new Zig-native HTTP engine that delivers 2x the JSON throughput of Node.js 22 on equivalent hardware.

The New JSON-First HTTP API

The biggest ergonomic win in Bun 1.2 is the Response.json() shorthand and a new req.json() body parser that together eliminate the boilerplate that has plagued REST API development in JavaScript runtimes.

Bun.serve({
  port: 3000,
  async fetch(req) {
    // Parse JSON body — one line, no middleware
    const body = await req.json();

    // Validate and respond
    if (!body.name || typeof body.name !== "string") {
      return Response.json(
        { error: "name is required", code: "MISSING_FIELD" },
        { status: 400 }
      );
    }

    return Response.json({
      id: crypto.randomUUID(),
      name: body.name,
      createdAt: new Date().toISOString()
    });
  }
});

Response.json() automatically sets Content-Type: application/json; charset=utf-8, serializes the object with JSON.stringify, and handles circular reference errors gracefully. No Express, no Hono, no middleware chain — just native JSON I/O.

Streaming JSON Responses

Bun 1.2 also introduces first-class streaming JSON support via ReadableStream integration. You can now stream large JSON arrays to clients without buffering the entire response in memory:

Bun.serve({
  async fetch(req) {
    const stream = new ReadableStream({
      async start(controller) {
        controller.enqueue('{"items":[');
        
        for await (const item of fetchLargeDataset()) {
          controller.enqueue(JSON.stringify(item) + ',');
        }
        
        controller.enqueue(']}');
        controller.close();
      }
    });

    return new Response(stream, {
      headers: { "Content-Type": "application/json" }
    });
  }
});

For APIs returning thousands of records, this pattern reduces time-to-first-byte by eliminating the full serialization step before the first byte is sent to the client.

JSON Schema Validation Integration

Alongside the new HTTP primitives, Bun 1.2 introduces a Bun.validateJSON() utility that bridges the JSON Schema ecosystem into the runtime itself. It works with any Draft 7 or 2020-12 schema and returns typed errors:

const schema = {
  type: "object",
  properties: {
    email: { type: "string", format: "email" },
    age: { type: "integer", minimum: 0, maximum: 150 }
  },
  required: ["email"]
};

const result = Bun.validateJSON(body, schema);

if (!result.valid) {
  console.log(result.errors);
  // [{ path: "/email", message: "must be a valid email address" }]
}

Migration from Express and Fastify

For teams moving existing Node.js JSON APIs to Bun 1.2, the runtime now ships a compatibility shim for Express-style req.body parsing. Most JSON APIs can be migrated with minimal changes:

  • Replace app.use(express.json()) with native await req.json()
  • Replace res.json(data) with return Response.json(data)
  • Remove body-parser, cors, and helmet — Bun handles these natively
  • Middleware chains map to async function composition

What's Next

The Bun roadmap for 1.3 includes HTTP/3 and QUIC support, native WebSocket JSON framing, and a built-in JSON query engine similar to jq — all compiled into the runtime at zero bundle cost. With Bun 1.2, the case for using it as a JSON API runtime has never been stronger.
