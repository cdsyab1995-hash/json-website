JSON Streaming APIs Go Mainstream: Browser-Native ReadableStream Lands in All Major Engines

Published: 2026-04-29 · 4-5 min read

A milestone for the web platform: as of early 2026, all three major browser engines — V8 (Chrome/Edge), SpiderMonkey (Firefox), and JavaScriptCore (Safari) — ship native streaming JSON parsing via the ReadableStream API. Developers can now consume large JSON API responses progressively without bundling third-party streaming parsers, unlocking faster perceived performance for data-heavy applications.

How Browser JSON Streaming Works

The browser-native approach uses the Fetch API's ReadableStream body combined with a streaming JSON parser exposed as a WHATWG Transform Stream:

const response = await fetch('https://api.example.com/large-dataset');

// Get the response body as a ReadableStream
const reader = response.body
  .pipeThrough(new TextDecoderStream())
  .pipeThrough(new JSONParseStream()) // New browser-native transform
  .getReader();

// Read JSON objects as they arrive
while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  renderItem(value); // Progressive rendering
}

JSONParseStream is the new browser-native transform stream that accepts a stream of text chunks and emits complete JSON values as they become parseable. For JSON arrays, each array element is emitted as a separate read event.

Server-Sent Events and NDJSON

The streaming JSON support pairs naturally with two popular API patterns:

  • NDJSON (Newline-Delimited JSON): Each line is a complete JSON object. JSONParseStream handles NDJSON out of the box, emitting one value per newline-terminated JSON string.
  • JSON arrays with incremental flushing: Servers that flush array elements before the closing bracket are now consumable progressively in the browser.

Why This Took So Long

Browser vendors have been cautious about adding opinionated JSON parsing to the web platform. The main concerns were: handling malformed JSON in a stream (should it throw on the first error or continue?), memory behavior for very large values, and the question of whether to support partial object streaming vs. only complete values. The final specification chose a conservative approach: JSONParseStream only emits complete, valid JSON values and throws a SyntaxError on malformed input.

Practical Impact

For developers building dashboards, feed UIs, or any interface that displays lists of API data, browser-native JSON streaming means the first items can appear on screen in hundreds of milliseconds rather than seconds. A product listing page fetching 500 items can show the first 20 while the rest are still in transit — without any framework or library changes, just the Fetch API and JSONParseStream.
