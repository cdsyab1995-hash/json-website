Newline-Delimited JSON (NDJSON) Gains Traction for Streaming

Published: 2026-05-05 · 6 min read · 1150 words

Why NDJSON Is Becoming the Go-To Format for Streaming Data

Traditional JSON uses arrays to represent collections of records. This design forces a fundamental tradeoff: you cannot parse element number five until you have received and validated elements one through four, because the parser must maintain the structural integrity of the containing array. For collections that fit comfortably in memory, this is a non-issue. But for datasets with millions of records — database exports, event logs, machine learning training sets, IoT sensor streams — the requirement to buffer the entire array before processing any element is prohibitive.

NDJSON (Newline-Delimited JSON, also called JSON Lines or ldjson) solves this with an elegant simplification: each line in the file or stream is a complete, standalone, valid JSON document. There are no enclosing arrays, no top-level structure requiring completion before parsing, and no possibility of a late parse error invalidating records already successfully processed. Receiving a newline means the current record is done; parse it, act on it, discard it, and move to the next line. Memory usage stays constant regardless of total dataset size.

The Format Specification: Deceptively Simple

The NDJSON specification (ndjson.org) is a single page, reflecting the format's philosophical simplicity. Each line must be a valid JSON value — typically an object but technically any JSON type. Lines are separated by newline characters (LF, \n). Carriage returns (CR) before the newline are tolerated for Windows compatibility. Blank lines are ignored. The file extension is .ndjson or .jsonl. The MIME type is application/x-ndjson.

This simplicity is both the format's strength and a source of occasional confusion. Because each line is independent, NDJSON has no concept of document-level metadata, schema declarations, or headers. The first line is a data record, not a schema. If schema information is needed, convention is to include a special object in the first line (e.g., a record with a $schema or _schema key), but this is not standardized. For production use, pairing NDJSON data files with a separate JSON Schema file is the recommended approach.

Real-World Use Cases Driving Adoption

The clearest evidence of NDJSON's growth is its adoption by major database and data engineering platforms. Elasticsearch uses NDJSON as the native format for its Bulk API — bulk-indexing millions of documents requires streaming, and NDJSON enables Elasticsearch to process records as they arrive rather than buffering an entire JSON array. The mongoexport and mongoimport tools use NDJSON by default. PostgreSQL's COPY command can read NDJSON. Snowflake and BigQuery both support NDJSON for streaming inserts and data loading.

In the observability space, log aggregation pipelines have embraced NDJSON because individual log events map perfectly to individual JSON lines. Logstash, Fluentd, and Fluentbit all produce and consume NDJSON natively. Cloud providers' logging services (AWS CloudWatch, Google Cloud Logging) use NDJSON for log exports. Machine learning teams use NDJSON for training data — the Hugging Face datasets library works natively with JSONL files, and many popular model fine-tuning formats like OpenAI's chat completion fine-tuning format use NDJSON.

Parsing NDJSON Efficiently Across Languages

Every major language has efficient NDJSON parsing options, and the basic pattern is identical across all of them: read lines one at a time and parse each line as JSON. In Python, the simplest approach is iterating over an open file and calling json.loads() per line. For files that are too large to fit in memory, the ijson library provides a higher-level streaming API. The ndjson Python package (pip install ndjson) wraps the standard json module with a clean line-by-line API that mirrors json.load() and json.dump().

In Node.js, the readline module combined with JSON.parse() handles streaming NDJSON from files or HTTP responses. The ndjson npm package provides transform streams that fit naturally into Node.js pipeline architecture. In Go, bufio.Scanner with json.Decoder processes NDJSON efficiently at high throughput. In Java, Jackson's MappingIterator from readValues() method reads NDJSON from any InputStream with minimal memory overhead. Rust users prefer the serde_json crate with a BufRead wrapper for line-by-line processing.

NDJSON vs. JSON Arrays: When to Use Each

The decision between NDJSON and standard JSON arrays should be driven by the characteristics of the data and how it will be consumed. JSON arrays are the right choice when: the entire collection is a meaningful semantic unit that must be validated as a whole; when consumers need random access to elements by index; when the collection is small enough to fit comfortably in the memory budget of the consuming process; or when interoperability with systems that only understand standard JSON is required.

NDJSON is the right choice when: processing must begin before all data has been received (streaming scenarios); when total data volume may exceed available memory; when records are independent and can be processed, validated, or stored individually; when fault tolerance requires that a parse error in one record does not invalidate all subsequent records; or when the data is being generated continuously (event streams, live log tails). Many modern data architectures use both: NDJSON for streaming ingestion and bulk processing, and JSON arrays for API responses where the client needs the complete collection to render a page.

Tools and Ecosystem for Working with NDJSON

The NDJSON tooling ecosystem has matured significantly alongside the format's adoption. jq, the standard JSON processing tool on Unix systems, works naturally with NDJSON: running jq '.' on an NDJSON file formats each line independently without modification. jq -s (slurp mode) reads all lines into an array when you need to work with the collection as a whole. The combination of jq and NDJSON files is one of the most productive data wrangling environments available in a terminal.

For data conversion, the csvjson and csvkit tools convert between CSV and NDJSON, handling common issues like header rows and type inference. The ndjson-cli npm package provides Unix-style filter, map, reduce, and join operations for NDJSON streams, enabling shell-script-style data pipelines for JavaScript developers. For validation, a simple wrapper around any JSON Schema validator processes NDJSON by applying the schema to each line independently, stopping on the first validation error or collecting all errors depending on the use case.

HTTP Streaming with NDJSON

NDJSON is increasingly used for HTTP APIs that need to stream results rather than buffer and return a complete response. The pattern is straightforward: the server sends the HTTP response headers with Content-Type: application/x-ndjson and Transfer-Encoding: chunked, then writes one JSON record per line as data becomes available, flushing after each newline. The client reads the response as a stream, parsing each line as it arrives.

This pattern is particularly valuable for long-running queries, real-time event feeds, and progressive loading of large datasets. The Kubernetes API uses NDJSON (with watch endpoints) to stream resource change events to kubectl and client libraries. GitHub's API uses NDJSON-like streaming for webhook event replays. OpenAI and Anthropic's chat completion APIs use server-sent events (SSE) with JSON payloads per event — a variation of the NDJSON streaming concept adapted for browser-native EventSource compatibility.

For frameworks, Express.js streams NDJSON via res.write(); FastAPI uses StreamingResponse; and Go's http.ResponseWriter.Write() handles it directly. The client-side implementation is equally simple: the Fetch API's response.body provides a ReadableStream, and a TextDecoder with line splitting gives you the individual JSON objects.
