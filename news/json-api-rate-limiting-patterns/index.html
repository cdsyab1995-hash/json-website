Rate Limiting JSON APIs: Patterns and Best Practices

Published: 2026-05-05 · 7 min read · 1310 words

Why Rate Limiting Is Non-Negotiable for JSON APIs

Rate limiting is the practice of restricting how many requests a client can make to an API in a given time window. It is not optional for any production API that matters. Without rate limiting, a single misbehaving client — whether through a bug in retry logic, a malicious actor attempting to exhaust your resources, or a legitimate user who accidentally wrote an infinite loop — can degrade service for every other client simultaneously. JSON APIs face particular risk because JSON is so easy to consume programmatically that clients can generate extremely high request rates with minimal code.

The consequences of inadequate rate limiting range from inconvenient (elevated infrastructure costs from runaway clients) to catastrophic (service outage affecting all users during a targeted abuse event). Building rate limiting in from day one is far easier than retrofitting it onto a live API with established clients — retrofitting requires coordinating behavior changes with all clients, managing migration periods, and handling the edge cases of clients that fail to respect new limits.

Token Bucket: The Most Flexible Algorithm

Token bucket is the most commonly used rate limiting algorithm in production APIs, and for good reason: it handles burst traffic gracefully while enforcing a sustained rate limit. The mental model is intuitive: each client has a bucket with a maximum capacity (the burst limit). Tokens accumulate in the bucket at a steady rate (the refill rate). Each API request consumes one token. If the bucket is empty, the request is rejected with a 429 Too Many Requests response. If the bucket has tokens, the request proceeds and one token is consumed.

The burst allowance is what makes token bucket appealing for real APIs. A client that has been idle accumulates tokens up to the bucket's maximum capacity, then can make that many requests quickly — modeling the natural behavior of applications that work in bursts rather than at perfectly constant rates. For example, a client with a burst limit of 100 and a refill rate of 10 per second can make 100 requests immediately after a period of inactivity, then sustains 10 requests per second indefinitely. This matches the actual traffic pattern of most legitimate clients better than algorithms that enforce perfectly constant rates.

Sliding Window: Precision at Scale

Fixed window rate limiting divides time into discrete windows (e.g., the current minute) and counts requests within each window. When the window resets, the counter resets to zero. This creates an exploitable edge case: a client can make the full limit of requests at the end of one window and immediately make the full limit again at the start of the next window, briefly achieving twice the nominal rate limit during the window boundary.

Sliding window rate limiting eliminates this boundary problem by maintaining a rolling time window rather than fixed segments. The implementation uses a sorted data structure (typically a Redis sorted set) that records the timestamp of each recent request. When a new request arrives, the algorithm removes all entries older than the window duration, counts the remaining entries, and rejects the request if the count exceeds the limit. This approach is more computationally expensive than fixed window counting, but the precise enforcement and absence of boundary exploitation make it the preferred algorithm for security-sensitive APIs.

Communicating Rate Limits via HTTP Headers

A rate limiting implementation that does not communicate its limits and current state to clients is only half-built. Clients need rate limit information to implement intelligent retry logic rather than retrying immediately on 429 responses and making the problem worse. The industry has converged on a set of standard HTTP headers for conveying rate limit information.

RateLimit-Limit (or the legacy X-RateLimit-Limit) communicates the maximum number of requests allowed in the time window. RateLimit-Remaining communicates how many requests the client can still make before hitting the limit. RateLimit-Reset communicates when the current window expires and the counter resets, typically as a Unix timestamp or a number of seconds. Retry-After is sent with 429 responses and tells the client exactly how many seconds to wait before retrying.

Sending these headers on every response — not just on 429 responses — allows clients to implement proactive rate limit awareness, slowing down before they hit the limit rather than reacting after the fact. This is a significant quality-of-life improvement for API consumers.

Distributed Rate Limiting with Redis

Rate limiting at scale requires shared state across all instances of your API service — otherwise, a client can bypass a per-instance limit by distributing requests across multiple instances. Redis is the standard choice for distributed rate limiting because it supports atomic operations that can implement any rate limiting algorithm correctly without race conditions.

For token bucket, the canonical Redis implementation uses a Lua script that atomically reads the current token count, calculates tokens accumulated since the last refill, updates the token count, and returns whether the request is allowed — all in a single Redis operation that cannot be interrupted by concurrent requests. For sliding window, a Redis sorted set stores request timestamps with ZADD, stale entries are removed with ZREMRANGEBYSCORE, and the current count is read with ZCARD. The entire operation runs as a single Lua script or a Redis pipeline for atomicity. Redis Cluster enables horizontal scaling of the rate limiting tier itself for very high-throughput APIs.

Rate Limiting Strategies for Different Client Types

A one-size-fits-all rate limit is rarely optimal. Different client types have legitimately different usage patterns, and a rate limit appropriate for individual users may be far too restrictive for an enterprise integration or far too permissive for anonymous access.

Authenticated individual users typically receive moderate limits appropriate for interactive application use: hundreds of requests per minute. Unauthenticated requests receive tighter limits (tens of requests per minute) because IP-based tracking is less reliable and unauthenticated access is more prone to abuse. API keys for automated integrations can receive higher limits based on the agreed-upon use case. Enterprise customers with SLAs may receive dedicated rate limit tiers with limits negotiated contractually. Bot detection based on request patterns — suspiciously regular intervals, no variation in User-Agent, unusual request sequences — can trigger additional restrictions beyond standard rate limits.

Handling Rate Limited Clients: Retry Logic Best Practices

From the client side, encountering a 429 response is not an error condition — it is information. A well-designed API client should treat rate limiting as feedback to slow down, not as an unexpected failure to report. The recommended client-side retry strategy is exponential backoff with jitter: wait a base delay, double it with each retry, and add random jitter to prevent multiple clients from synchronizing their retries and creating periodic traffic spikes.

The Retry-After header from the API is more precise than exponential backoff guessing — when it is present, the client should respect it exactly rather than applying its own backoff logic. For high-throughput batch processing jobs, proactive rate limit management is more efficient: read RateLimit-Remaining from each response, and when it drops below a threshold, introduce deliberate sleeps to stay within the limit rather than hitting 429 responses and backing off.

Rate Limiting at the Edge: CDN and Gateway Patterns

For APIs that face the public internet, implementing rate limiting at the application layer (in your API server code) is the last line of defense, not the first. Moving rate limiting to the edge — to a CDN, API gateway, or reverse proxy — filters abusive traffic before it reaches application servers, saving compute resources and reducing the blast radius of traffic spikes.

Cloudflare's rate limiting rules can enforce limits by IP, by cookie, by custom request headers, or by any combination, with no application code changes required. AWS API Gateway has built-in usage plans and API keys that enforce per-key limits at the AWS network edge. Kong and Tyk are open-source API gateways with sophisticated rate limiting plugins that support all major algorithms. Nginx and Envoy Proxy can implement token bucket and sliding window limits via their rate limiting modules. For most public APIs, a multi-layer approach is optimal: edge-level IP rate limiting for obvious abuse, gateway-level key-based limits for authenticated traffic, and application-level limits for fine-grained per-resource control.
