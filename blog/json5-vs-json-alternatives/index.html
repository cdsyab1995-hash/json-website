JSON5 vs JSON: When to Use Human-Friendly Syntax

Published: 2026-05-05 · 9 min read · 1866 words

Introduction: The Problem with Writing JSON by Hand

JSON was designed for machine-to-machine communication, not for humans to write. This design choice shows clearly when you sit down to create a configuration file or write test fixtures by hand. You constantly forget whether to use single or double quotes. You accidentally add trailing commas. You desperately want to add a comment explaining why a value is set to something non-obvious. And every time, JSON's strict syntax rejects your human instincts.

JSON5 was created to solve exactly these problems. It is a superset of JSON that adds features JavaScript developers already know, making it far more comfortable for human authoring while remaining fully machine-parseable. In this guide, we compare JSON5 with standard JSON, explore other JSON alternatives, and help you decide which format fits your use case.

What JSON5 Adds: Every Enhancement Explained

JSON5 makes six key improvements over standard JSON. Comments: both single-line (//) and multi-line (/* */) comments are allowed anywhere. Trailing commas: arrays and objects can end with a comma after the last element—a constant pain point when editing JSON. Single quotes: strings can use single quotes instead of double quotes, matching common JavaScript style. Unquoted property names: {foo: 1} is valid JSON5 if the key is a valid identifier. Hex numbers: 0xFF and similar hex literals are valid. Multiline strings: a string can span multiple lines using a backslash at the end of each line.

Each of these additions addresses a specific frustration with standard JSON while staying close to JavaScript syntax that most web developers already know. The result is a format that feels natural to write and still parses unambiguously.

When JSON5 Makes Sense

JSON5 shines in configuration files that humans read and edit. If you have written a package.json, tsconfig.json, .babelrc, or .eslintrc, you have dealt with JSON that teams need to understand and modify. The ability to add comments is particularly valuable: you can explain why a setting is set to a specific value, document the range of acceptable values, or warn about known issues.

JSON5 is also excellent for test fixtures, example data, and seed data where comments can explain the purpose of each record. For machine-to-machine API responses, however, standard JSON is always the right choice—the performance overhead of JSON5 parsing and the lack of universal support make it inappropriate for high-volume data interchange.

Parsing JSON5 in Major Programming Languages

The json5 npm package provides JSON5.parse() and JSON5.stringify() functions that mirror the standard JSON API. Installation is simple: npm install json5. Then import json5 from 'json5' and replace JSON.parse with json5.parse. For Node.js configuration loading: const config = json5.parse(fs.readFileSync('config.json5', 'utf8')).

In Python, the pyjson5 library provides similar functionality: import pyjson5; config = pyjson5.load(open('config.json5')). For Java, the json5-java library supports JSON5 parsing. Most language ecosystems have at least one JSON5 implementation. When choosing a library, check activity level, test coverage, and spec compliance. Some implementations have subtle differences in handling edge cases.

JSON5 in Practice: Real Configuration Files

Several major tools already use JSON5 or JSON5-like formats. TypeScript's tsconfig.json is technically JSON5—it allows comments and trailing commas. The VS Code settings.json file allows comments. Babel's .babelrc supports comments. These tools chose JSON5 because configuration files are written and maintained by humans, and the quality-of-life improvements make the files easier to work with.

Converting an existing JSON configuration to JSON5 is straightforward: rename the file from .json to .json5, update the parser to use json5.parse(), and start adding comments. The content does not need to change because JSON5 is a superset of JSON—all valid JSON is valid JSON5. This backward compatibility makes migration risk-free.

Limitations and Trade-offs of JSON5

JSON5 has real limitations to consider. Universal support: JSON5 parsers exist in most languages, but they are third-party libraries, not built-in. Standard JSON is natively supported everywhere without dependencies. Specification stability: JSON5 has been stable since version 1.0, but the spec does not have the same weight as RFC 8259 (standard JSON). Tooling: some JSON-aware tools (validators, formatters, diff tools) do not understand JSON5, requiring workarounds.

Performance is slightly lower due to the added grammar complexity, though for configuration files this is completely irrelevant—parsing happens once at startup and takes milliseconds even for large files. The practical impact of these trade-offs depends on your context: for configuration files used by a team of developers, JSON5 limitations are minor compared to its benefits.

YAML: The Configuration File Heavyweight

YAML (YAML Ain't Markup Language) is the most widely used alternative to JSON for configuration files. It emphasizes human readability with minimal punctuation. YAML supports comments, multiline strings, references (anchors and aliases), and includes for splitting configuration across files. It is the configuration format of choice for Kubernetes, GitHub Actions, Docker Compose, and many other DevOps tools.

The appeal of YAML is its clean syntax: key: value pairs without quotes, indentation to show structure, and natural list syntax with dashes. However, YAML has a dangerous dark side: its significant whitespace means a misplaced space can silently change behavior. The spec is enormous and complex, with many surprising edge cases. The infamous Norway problem (where the two-letter country code NO parsed as boolean false) illustrates how subtle YAML can be. For simple configuration, YAML is pleasant; for complex configuration maintained by large teams, its footguns can cause production incidents.

TOML: The Opinionated Configuration Language

TOML (Tom's Obvious Minimal Language) was created as a reaction against YAML's complexity. It uses an INI-like syntax that is immediately familiar, unambiguous, and easy to implement. TOML is the configuration format for Rust's Cargo, Python's Poetry and Pyproject.toml, and Hugo static site generator.

TOML's key features: sections with [table] headers, arrays with [[array of tables]] syntax, inline tables and arrays, and support for dates, times, and integers natively. The spec is deliberately small and clear, making implementations consistent across languages. The main limitation is poor support for deeply nested structures—TOML gets verbose for complex hierarchical configuration. For projects with moderate configuration complexity, TOML is an excellent choice.

HJSON: Human JSON for Configuration

HJSON (Human JSON) is another JSON extension focused on configuration file use. It goes further than JSON5 with more relaxed string handling: strings do not require quotes if they do not contain special characters. Multi-line strings use triple quotes. Comments are allowed. Trailing commas are optional.

HJSON prioritizes writability even more than JSON5. The trade-off is less compatibility: HJSON looks less like JSON than JSON5 does, which can be disorienting for developers familiar with JSON. The ecosystem is smaller than JSON5. HJSON is a niche choice that makes sense when human authoring comfort is paramount and you are willing to invest in tooling to support it. For most projects, JSON5 or TOML are better options.

HOCON: Hierarchical Configuration Format

HOCON (Human-Optimized Config Object Notation) is used by the Play Framework, Akka, and Lightbend's configuration library. It adds powerful features beyond comments and trailing commas: file includes, variable substitution, object merging, and path expressions. A HOCON file can reference environment variables directly: apiKey = ${?API_KEY}.

HOCON's power comes at the cost of complexity. The spec is extensive, and the format is specific to JVM-based tooling. It is excellent for complex applications with environment-specific configuration overlays, hierarchical defaults, and cross-file references. For simpler needs, this power is overkill. HOCON is a professional tool for professional configuration management challenges.

JSON Lines (JSONL): JSON for Streaming and Logs

JSON Lines (also called NDJSON—Newline Delimited JSON) is not a JSON alternative for configuration—it is a format for streaming data. Each line is a complete, valid JSON value (usually an object). No enclosing array, no commas between records, just one JSON value per line.

JSON Lines is ideal for log files, event streams, and data exports. You can append records without reading and rewriting the entire file. You can process records one at a time without loading the entire file into memory. Grep and simple text tools work well on JSONL files. The tradeoff is that files are not valid JSON—you cannot open a JSONL file in a JSON viewer and see structured data. For data pipelines processing millions of records, JSONL's streaming advantages are compelling.

MessagePack and CBOR: Binary JSON Alternatives

When JSON's text format becomes a performance bottleneck, binary alternatives offer significant improvements. MessagePack is a binary serialization format that maps closely to JSON's data model but encodes data 20-30% smaller and parses significantly faster. It is widely used in gaming, finance, and high-frequency trading where every microsecond counts.

CBOR (Concise Binary Object Representation) is a RFC 7049 standard binary format based on JSON's data model. It supports additional types including binary data, large integers, and dates without string encoding. CBOR is used in IoT devices, CoAP protocols, and COSE (CBOR Object Signing and Encryption). Both MessagePack and CBOR sacrifice human readability for performance and size—the right trade-off for machine-to-machine communication where performance matters more than debuggability.

Choosing the Right Format: A Decision Framework

Choosing between JSON and its alternatives comes down to four questions. Who writes it: humans or machines? If humans, consider JSON5, YAML, TOML, or HOCON. If machines, use standard JSON or a binary format. What matters more: readability or performance? Readability points to JSON5/YAML; performance points to MessagePack/CBOR. What is the ecosystem: which formats are supported by your tools and team? Familiar formats reduce friction. What are the stakes: configuration files that rarely change can use more exotic formats; high-traffic API payloads should use well-supported standards.

For most web applications: API responses use standard JSON, configuration files use JSON5 or TOML, logs use JSON Lines, and internal high-performance services optionally use MessagePack. This pragmatic approach uses the best tool for each job.

Migration Guide: Moving from JSON to JSON5

Migrating an existing JSON configuration to JSON5 is risk-free and quick. Step 1: Rename .json files to .json5 (or keep both and phase out JSON gradually). Step 2: Update file loading code to use a JSON5 parser instead of JSON.parse. Step 3: Verify that the JSON5 parser correctly reads the existing content—no changes should be needed since JSON is valid JSON5. Step 4: Start adding comments to explain non-obvious settings. Step 5: Gradually adopt JSON5 features like trailing commas in new additions.

The migration is fully reversible: JSON5 files can be converted back to standard JSON by stripping comments and trailing commas. This makes JSON5 a low-risk experiment you can adopt without permanent commitment.

JSON5 in Build Tools and Configuration

The JavaScript and TypeScript ecosystem has effectively standardized on JSON5 for configuration files, even if they do not call it that. TypeScript's tsconfig.json allows comments and trailing commas—these are JSON5 features. VS Code's settings.json allows comments. Babel, ESLint, and Prettier all support configuration files with comments.

For new projects, consider adopting JSON5 for all configuration files from the start. Name configuration files with .json5 extensions to make the format explicit and enable proper syntax highlighting. Configure your IDE to recognize .json5 files with JSON5 language support for proper highlighting and validation. This small investment pays dividends in configuration readability throughout the project's lifetime.

Conclusion: Matching Format to Purpose

The JSON alternatives landscape offers genuinely useful options for different scenarios. JSON5 improves human authoring with minimal complexity overhead. YAML provides maximum flexibility with real footguns. TOML offers clarity and simplicity for moderate-complexity configuration. HOCON adds power for complex hierarchical configuration. JSON Lines enables streaming data. MessagePack and CBOR optimize for binary performance.

Standard JSON remains the king for API responses, data interchange, and any context where universal support and simplicity matter. It is not the best format for every job, but it is adequate for almost all of them and perfect for machine-to-machine communication. Choose JSON5 or TOML when humans need to read and edit configuration files. Use standard JSON everywhere else. This simple guideline covers the vast majority of real-world decisions.
