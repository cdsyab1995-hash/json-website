JSON Structured Logging: Best Practices for Production Applications

Published: 2026-05-05 · 7 min read · 1346 words

Why Structured Logging Transforms Observability

Traditional logging writes human-readable messages to text files: 'User 123 logged in successfully', 'Payment processing failed for order ABC'. These messages are easy for developers to read during debugging but become unwieldy at scale. Searching for specific events across millions of log lines requires fragile regex patterns. Counting events requires parsing unstructured text. Correlating events across different log files is nearly impossible.

Structured logging addresses these problems by writing machine-readable log entries — typically as JSON — where each piece of information is a named field. Instead of parsing 'User 123 logged in successfully' with a regex to extract the user ID, the log entry contains an explicit user_id field. This makes logs queryable, aggregatable, and joinable using standard data processing tools.

This guide covers the implementation of structured JSON logging across different programming languages, best practices for log entry structure, and integration patterns with popular log aggregation and analysis platforms.

Log Entry Structure and Field Naming

A well-structured JSON log entry contains a standard set of fields that every log entry should include, plus domain-specific fields relevant to the specific event being logged. The standard fields ensure that logs can be processed by generic log aggregation and analysis tools.

The essential standard fields are: timestamp (ISO 8601 format in UTC, not local time), log level (DEBUG, INFO, WARN, ERROR, CRITICAL in uppercase), message (human-readable description of the event), and a correlation identifier (request ID or trace ID that enables correlating logs across different services in a distributed system). Beyond these, the most commonly useful additional fields are service name (for multi-service systems), environment (production, staging), and host or container identifier.

Domain-specific fields should capture the data relevant to the specific event type. A user authentication log entry should include user_id, authentication_method, and success/failure status. A payment processing log entry should include order_id, amount, currency, and payment_provider. The goal is to capture enough context that a developer debugging an issue from logs alone can understand exactly what happened without needing to reproduce the scenario.

Log Levels and What to Actually Log

Structured logging makes it practical to emit logs at any level for any event, which creates a new challenge: deciding what to log. Excessive logging creates noise that obscures important events and increases storage costs. Insufficient logging leaves gaps that make debugging difficult.

The DEBUG level should capture detailed diagnostic information: entering and exiting functions, values of intermediate variables, retry attempts, and cache hits or misses. DEBUG logs are typically disabled in production unless actively troubleshooting a specific issue. The INFO level should capture significant business events: user actions, business transactions, system state changes. Every INFO log entry should represent something a non-developer (product manager, support engineer) would find meaningful.

WARN logs capture situations that are unexpected or potentially problematic but do not prevent the operation from completing. Examples: retry succeeded after initial failure, cache miss for expensive operation, deprecated API used. ERROR logs capture operations that failed and required alternative handling or resulted in a user-visible failure. CRITICAL logs capture systemic failures that affect multiple users or services. The key principle is that every log entry should be answerable to the question: 'If I saw this log entry in a dashboard, would I know what to do?'

Structured Logging in Python

Python's standard logging module supports structured logging through the StructLog library, which has become the de facto standard for structured logging in Python applications. StructLog provides a clean, declarative API for creating structured log entries.

Basic usage involves binding context variables that persist across a request and formatting the output as JSON. The bound context variables (like request_id, user_id, and correlation_id) are automatically included in every log entry during the request lifecycle, eliminating the tedium of manually including these values in every log call. The json method configures StructLog to output JSON, which can then be piped to any log aggregation system.

For FastAPI and other async frameworks, StructLog's processor chain integrates with the framework's middleware system. You can inject a request_id at the middleware level, and StructLog automatically propagates it through the async call graph. This ensures that every log entry during a request carries the same request_id, making it trivial to retrieve all logs for a specific user action.

Structured Logging in JavaScript and TypeScript

JavaScript and TypeScript applications benefit from pino, widely regarded as the fastest structured logging library for Node.js. Pino's design philosophy prioritizes performance — it produces JSON output with minimal processing overhead, making it suitable for high-throughput production services.

Pino's child logger feature provides context propagation similar to StructLog's bound variables. You create a child logger with shared context (request_id, user_id) and use that child logger throughout the request lifecycle. Each log entry automatically includes the shared context, and the overhead compared to a bare log call is negligible — pino achieves this through a clever approach of pre-serializing the shared context.

For TypeScript applications, pino-ts (a type-safe wrapper around pino) provides full TypeScript type inference for log entry fields. This means that if you log a field with the wrong type, the TypeScript compiler catches it at compile time rather than discovering the type mismatch in production logs.

Log Aggregation and Analysis Platforms

Structured JSON logs are valuable only when they can be efficiently queried and analyzed. The log aggregation platform you choose determines what you can do with your logs. Elasticsearch, Logstash, and Kibana (the ELK stack) is the most common open-source combination. Logstash ingests JSON logs, Elasticsearch stores and indexes them, and Kibana provides a visualization and query interface.

Cloud-native alternatives like AWS CloudWatch Logs, Google Cloud Logging, and Azure Monitor offer native JSON log ingestion with built-in query capabilities. These services integrate tightly with their respective cloud platforms, making them convenient choices for cloud-deployed applications. The query languages vary significantly — CloudWatch Logs Insights uses its own DSL, while Google Cloud Logging supports both a simple text search and a more powerful structured query language.

For teams needing advanced log analysis and correlation, commercial platforms like Datadog, New Relic, and Grafana Loki offer powerful structured log querying with integrations across infrastructure, application performance, and business metrics. The correlation capabilities across these dimensions are particularly valuable for understanding the full impact of issues — from the application error that caused a performance degradation to the business outcome it produced.

Sampling Strategies for High-Volume Logging

Even with structured logging's efficiency, high-volume applications can generate millions of log entries per second. Sampling — logging only a representative fraction of events — reduces volume while preserving statistical patterns and enabling detection of rare events.

Deterministic sampling (also called consistent sampling) ensures that all log entries for the same request always include or exclude the sampled entry. This is implemented by hashing a stable identifier (like the request ID) and comparing against a threshold. All events for a given request either appear in full detail or are sampled out together, ensuring that debugging a specific request always has complete information.

Adaptive sampling adjusts the sampling rate based on system load. During normal operation, you might sample 10% of requests; during an incident investigation, you disable sampling to capture full detail. This approach requires instrumentation that detects when an incident is in progress — typically through elevated error rates — and adjusts the sampling configuration dynamically.

Security and Sensitive Data in Logs

Structured logging creates a new attack surface for data leaks: sensitive data that is logged inadvertently. Credit card numbers, passwords, authentication tokens, and personally identifiable information (PII) must never appear in logs. A single log entry containing a password in plaintext can compromise an entire system.

The solution is sanitization: transforming log entries to remove or mask sensitive fields before they are written. JSON log sanitizers can identify sensitive fields by name (fields containing 'password', 'secret', 'token', 'key', 'ssn', 'credit_card') and either remove them entirely or replace them with a placeholder like '[REDACTED]'. For structured logs with consistent field naming, this approach catches the majority of inadvertent sensitive data logging.

More sophisticated sanitization uses data classification to identify sensitive content regardless of field name. A credit card number detector that uses the Luhn algorithm can identify and redact credit card numbers even if they appear in a field named 'notes'. HIPAA and PCI-DSS compliance requirements often mandate specific sanitization patterns for healthcare and payment data respectively.
