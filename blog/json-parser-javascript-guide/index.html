Build Your Own JSON Parser in JavaScript: A Complete Guide

Published: 2026-05-05 · 16 min read · 3185 words

Introduction: Why Build Your Own JSON Parser?

You use JSON.parse() every day, but have you ever wondered what happens under the hood? Building your own JSON parser is one of the best ways to deepen your understanding of parsing theory, state machines, and recursive descent algorithms. The standard JSON.parse() is highly optimized and handles every edge case defined in RFC 8259, but it remains a black box. When it throws a cryptic 'Unexpected token' error, you have no insight into what went wrong or how to fix it. By building your own parser, you will understand exactly how JSON syntax is validated, how errors are detected, and how the parse tree is constructed. This knowledge translates to better debugging skills and a deeper appreciation for the tools you use daily.

Building a JSON parser is also a fantastic interview preparation exercise that tests your ability to work with strings, manage state, handle recursion, and think about edge cases. It requires careful planning, systematic testing, and iterative refinement—the same skills you need for production code. In this comprehensive guide, we will build a complete JSON parser in JavaScript from scratch, handling all standard cases and producing helpful error messages. By the end, you will have a working parser you can extend and customize.

Understanding JSON Grammar: The Foundation of Parsing

Before writing any code, you need to understand the JSON grammar precisely. According to RFC 8259, a JSON text is a serialized value, which can be one of six types: object, array, string, number, boolean, or null. Objects are unordered collections of name-value pairs enclosed in curly braces. Arrays are ordered lists of values enclosed in square brackets. Strings are sequences of Unicode characters wrapped in double quotes, with support for escape sequences like \n for newline, \t for tab, and \\ for backslash. Numbers follow a specific syntax: optional minus sign, integer part, optional decimal fraction, and optional exponent.

The simplicity of JSON grammar is deceptive. While the basic rules are straightforward, the edge cases are numerous and crucial. Strings can contain any Unicode character, including surrogate pairs for emoji and CJK characters. Numbers have multiple valid representations (1.0, 1e0, 1e+0). Unicode escape sequences can encode any character. The grammar is also strict: no trailing commas, no comments, no unquoted property names, no bare words (true, false, null must be lowercase). Understanding these rules completely is essential before you write a single line of parser code.

Architecture Overview: Tokenizer and Parser

A JSON parser has two main components: the tokenizer (lexer) and the parser itself. The tokenizer reads the raw input string and converts it into a sequence of tokens—meaningful units like numbers, strings, braces, and keywords. The parser reads these tokens and constructs the JavaScript object structure. This separation of concerns makes each component easier to understand and test.

The tokenizer operates character by character, building up tokens based on what it encounters. When it sees a double quote, it knows a string is coming and reads until the matching closing quote. When it sees a digit or minus sign, it reads the complete number syntax. When it sees a letter, it checks if it matches true, false, or null. The tokenizer also tracks whitespace and simply skips over it, since JSON allows arbitrary whitespace between any two tokens.

The parser then consumes these tokens according to the JSON grammar rules. It uses a recursive descent approach: parseValue() is the main entry point, and it dispatches to specialized functions like parseObject(), parseArray(), parseString(), and parseNumber() based on the current token type. Each function handles its specific grammar rule and recursively calls parseValue() for nested structures.

Building the Tokenizer: Character-by-Character Analysis

Let us start building the tokenizer. We will create a class called JSONTokenizer that takes an input string and provides a nextToken() method. The tokenizer maintains an internal position that tracks where we are in the input string. The nextToken() method advances past whitespace, then examines the current character to determine what type of token we are looking at.

If the current character is {, }, [, ], :, or ,, we return the corresponding token type immediately. If it is a digit or minus sign, we read the complete number token by consuming all valid number characters and returning a NUMBER token with the string value. If it is a double quote, we enter string-reading mode: we advance past the opening quote, then read characters one by one until we find the closing quote, handling escape sequences along the way. If it is t, f, or n, we check if the next characters match true, false, or null and return the appropriate token.

A crucial part of tokenization is tracking errors. If we encounter an unexpected character, we should report the position and a helpful error message. We should also handle end-of-input gracefully—if we reach the end while still expecting more tokens, that is an error too.

Handling Unicode Strings: Beyond ASCII

String tokenization is the most complex part of a JSON tokenizer because JSON supports the full Unicode character set. JavaScript handles Unicode natively through its string type, but you need to be careful about escape sequences. JSON supports several escape sequences: \\/ for forward slash, \\\\ for backslash, \\" for double quote, \\b for backspace, \\f for form feed, \\n for newline, \\r for carriage return, \\t for tab, and \\uXXXX for any Unicode character specified by its four-digit hexadecimal code.

When tokenizing a string, you need to handle these escape sequences specially. When you encounter a backslash, look at the next character to determine what escape sequence it is. If it is u, read the next four hex digits and convert them to the corresponding Unicode character. For other escape sequences, replace them with their single-character equivalents. Also, you must detect and report errors: an unclosed string (reached end-of-input before finding the closing quote), an invalid escape sequence (like \\x), or a lone surrogate (an incomplete Unicode pair) are all errors.

Modern JavaScript makes this easier because strings natively support Unicode. You can use template literals with \\u{} syntax for Unicode code points beyond BMP, or use String.fromCodePoint() to construct characters from code point numbers.

Recursive Descent Parsing: Theory and Implementation

With tokens in hand, we can build the parser using recursive descent. The idea is elegant: each grammar rule becomes a function, and functions call each other recursively to handle nested structures. The parseValue() function is the heart of the parser—it looks at the current token and dispatches to the appropriate sub-parser.

When parseValue() sees a LEFT_BRACE token, it knows we have an object and calls parseObject(). parseObject() expects a sequence of key-value pairs separated by commas, ending with RIGHT_BRACE. For each pair, it calls parseString() to get the key, expects a COLON token, then calls parseValue() to get the value. The beauty of recursive descent is that parseValue() can be called for any value, including nested objects and arrays, so arbitrary nesting depth is handled naturally.

parseArray() works similarly: it expects LEFT_BRACKET, then a (possibly empty) sequence of values separated by commas, ending with RIGHT_BRACKET. parseString(), parseNumber(), parseBoolean(), and parseNull() are leaf functions that simply consume their respective tokens and return the corresponding JavaScript values.

Error Handling: From Cryptic to Helpful

A parser that gives unhelpful error messages is nearly useless for debugging. When JSON.parse() gives you 'Unexpected token at position 42', it tells you almost nothing. A better parser would show you the problematic area, explain what was expected, and suggest what might be wrong.

To produce helpful errors, you need to track position information throughout parsing. Store the starting position of each token during tokenization. When you detect an error in the parser, use the position to extract the surrounding context from the input. Show a snippet of the input with a caret pointing to the error location. Explain what token was found and what was expected.

Common errors include: mismatched brackets (found } when expecting ]), missing commas between array or object elements, trailing commas after the last element, unquoted property names, invalid escape sequences in strings, numbers that are malformed (like 1.2.3), and unexpected end-of-input. For each error, craft a message that helps the user understand and fix the problem.

Building a Complete Token Class

Let us structure our tokenizer properly with a Token class and an enumeration of token types. A token has a type, a value, and position information (line and column numbers, or a character offset). The token types are: LEFT_BRACE ({), RIGHT_BRACE (}), LEFT_BRACKET ([), RIGHT_BRACKET (]), COLON (:), COMMA (,), STRING, NUMBER, TRUE, FALSE, NULL, and END_OF_INPUT.

The tokenizer should be an iterator or have a nextToken() method that returns the next token. It should also have a peek() method that returns the next token without consuming it—this is useful in the parser when you need to look ahead to decide what to do. The parser can maintain a currentToken variable and call advance() to move to the next token, which handles both consuming the current token and fetching the next one.

Here is a sketch of the structure: class Token { constructor(type, value, pos) { this.type = type; this.value = value; this.pos = pos; } } class JSONTokenizer { constructor(input) { this.input = input; this.pos = 0; } nextToken() { ... } } class JSONParser { constructor(tokenizer) { this.tokenizer = tokenizer; this.current = null; } advance() { this.current = this.tokenizer.nextToken(); } parse() { ... } }

Testing Your Parser: Building a Test Suite

Thorough testing is essential for a parser. You need tests for every token type, every valid JSON construct, and as many invalid inputs as possible. Start with the basics: parse 'null', 'true', 'false', empty object {}, empty array [], simple strings, simple numbers. Then move to nested structures: arrays of arrays, objects with object values, deeply nested structures.

Test Unicode handling: strings with emoji, CJK characters, and special symbols. Test escape sequences: all standard escapes and invalid escapes. Test numbers: integers, decimals, scientific notation, negative zero, large numbers. Test whitespace handling: tabs, newlines, multiple spaces, weird spacing.

For error cases, write tests that expect specific errors. Verify that your parser rejects trailing commas, missing commas, mismatched brackets, unclosed strings, invalid escapes, and other malformed inputs. Use a test framework like Jest to organize your tests and get clear pass/fail reporting. As you discover bugs, add regression tests to prevent them from reoccurring.

Performance Optimization: From Naive to Efficient

Our initial parser will work but may be slow for large inputs. A naive string-concatenation approach for building up string tokens can be O(n^2) for strings with many escape sequences. Tokenization character by character is inherently O(n), but object and array construction can also add overhead.

One optimization is to avoid creating intermediate strings. When tokenizing a string, instead of concatenating characters, you can track start and end indices and slice the original input. This works well for strings without escape sequences. For strings with escapes, you still need to process them, but you can build up the result in an array and join at the end.

Another optimization is in parsing: avoid creating unnecessary objects for trivial arrays and objects. Instead of building up an array element by element, count the elements first, create an array of the right size, then fill it. Similarly, pre-allocate object property storage. These micro-optimizations add up for large JSON documents.

Adding the Reviver Parameter: Advanced JSON.parse() Feature

The real JSON.parse() accepts a second parameter: a reviver function. This function is called for every value in the parsed result, allowing you to transform or filter the data during parsing. Our parser can support this feature too.

The reviver is called with the key (property name for objects, array index for arrays, empty string for the root) and the value. It can return a transformed value, return undefined to delete the property, or return something else entirely. To implement this, after parsing the complete JSON structure, call a reviveValue() function on the root with an empty key. This function recursively walks the structure, calling the reviver for each value and filtering out any undefined results.

The reviver feature is powerful for data transformation. You can rename keys, convert date strings to Date objects, filter out sensitive fields, or convert specific values based on their context. Adding this feature to your custom parser demonstrates deep understanding of both the parsing process and JavaScript object manipulation.

Handling Circular References: A Danger Zone

Standard JSON cannot represent circular references—objects that contain themselves directly or indirectly. Attempting to serialize circular references with JSON.stringify() throws an error. Your custom parser should be aware of this limitation and handle it gracefully if you extend it to support JSON supersets.

Circular references are dangerous because they can cause infinite loops during serialization or traversal. If you add a stringify() method to your parser, you must track visited objects to detect cycles. Use a WeakSet to keep track of objects you have seen; when you enter an object, add it to the set, recurse into its properties, then remove it from the set. If you encounter an object that is already in the set, throw a circular reference error.

In parsing, circular references are not a concern since you build the object graph fresh. But if you add any traversal or transformation features, circular references become a real danger. Always document the limitation clearly and test for it explicitly.

Comparison: Our Parser vs JSON.parse()

How does our custom parser compare to the native JSON.parse()? The native implementation is written in the JavaScript engine (likely C++ or Rust) and is highly optimized. It uses techniques like JIT compilation, native string handling, and direct memory access that JavaScript cannot match. For typical JSON documents, JSON.parse() is orders of magnitude faster than a pure JavaScript implementation.

However, our parser has advantages. It is customizable—you can add features, modify error messages, or adapt it for specific use cases. It is educational—you learn the internals of parsing. It can be extended with non-standard features like comments, trailing commas, or custom data types. And for small documents or educational purposes, the performance difference is negligible.

In production, use JSON.parse() for performance-critical code. But keep your custom parser for debugging, testing, and learning. Understanding how it works makes you a better JavaScript developer.

Extension Ideas: Beyond Standard JSON

Once you have a working JSON parser, you can extend it to support useful features that standard JSON lacks. Comments are the most requested feature—many developers want to include comments in their JSON configuration files. Supporting comments is straightforward: during tokenization, when you see //, skip everything until the end of the line; when you see /*, skip until */.

Trailing commas are another popular extension. Standard JSON does not allow a comma after the last element in an array or object, but JavaScript does. You can modify your parser to accept trailing commas, reducing frustration when editing JSON by hand.

You could also add support for single-quoted strings (like JavaScript), unquoted property names (like JavaScript object literals), hexadecimal numbers (like 0xFF), or special values like undefined and Infinity. Each extension teaches you something new about parser design and gives you a useful tool for specific use cases.

Debugging Techniques: When Your Parser Fails

When your parser fails, debugging requires a systematic approach. Start by isolating the problem: can you parse a minimal valid input? If not, the problem is in basic tokenization or parsing. Add logging to your tokenizer to see what tokens it produces. Check that the tokens are in the expected order and that END_OF_INPUT is reached at the right time.

If simple inputs work but complex ones fail, the problem is likely in recursive handling of nested structures. Add depth tracking to your parse functions and log entry and exit from each function. Look for mismatches: entering parseObject() five times but exiting four times indicates a missing return or premature exit.

For string parsing issues, log the exact characters being processed. Use JSON.stringify() on the input to see control characters and escape sequences clearly. Check for off-by-one errors in string boundaries—many string bugs come from including or excluding the wrong quote character.

JSON Parser in Node.js: Command-Line Tool

You can wrap your parser in a command-line tool for convenient use. Create a Node.js script that reads a file or stdin, passes the content to your parser, and outputs the result. Add error handling to catch and display parser errors gracefully. You can also add options like --pretty for formatted output, --validate for validation only (no output), and --reviver for custom transformation.

To make it globally available, add a shebang line (#!/usr/bin/env node), mark the file as executable, and either add it to PATH or use npm link. You can also publish it to npm so others can install and use your parser. Even if you do not publish it, having a CLI tool is useful for quick JSON validation and debugging.

Here is a minimal CLI implementation: #!/usr/bin/env node const fs = require('fs'); const input = fs.readFileSync(process.argv[2] || 0, 'utf8'); const tokenizer = new JSONTokenizer(input); const parser = new JSONParser(tokenizer); const result = parser.parse(); console.log(JSON.stringify(result, null, 2));

Security Considerations: Parser Security Risks

Parsers can have security vulnerabilities if not written carefully. The 'billion laughs' attack uses deeply nested JSON to exhaust memory and crash the parser. Standard JSON parsers set a maximum depth limit to prevent this. Your custom parser should do the same: track nesting depth and throw an error if it exceeds a reasonable limit (like 1000 levels).

Another risk is long strings causing excessive memory allocation. If a string token is gigabytes long, parsing it could consume all available memory. Set a maximum string length and reject inputs with strings exceeding that limit. Similarly, very long arrays or objects with many elements can cause issues.

If you add features like date parsing (converting ISO date strings to Date objects), be careful about code injection. The reviver function, if exposed to untrusted input, could execute arbitrary code. Always validate and sanitize any data that affects program flow or memory allocation.

Real-World Applications of Custom Parsers

Custom JSON parsers are not just academic exercises—they have many real-world applications. Configuration file parsing often requires JSON with extensions like comments. If your project uses JSON for configuration, a custom parser can support comments, variables, includes, and other convenience features.

Data transformation pipelines often need to parse, transform, and re-serialize JSON. A custom parser with a built-in reviver can perform transformations in a single pass, which is more efficient than parse-transform-serialize. You can also add validation rules beyond JSON Schema.

Legacy systems may use JSON-like formats that are not quite standard JSON. A custom parser can handle these formats directly, converting them to standard JSON for downstream processing. This is common when migrating from older systems or integrating with third-party services that use slightly non-standard JSON.

In this comprehensive guide, we have built a complete JSON parser in JavaScript from scratch. We covered the JSON grammar, tokenizer design, recursive descent parsing, error handling, Unicode handling, testing, performance optimization, and extensions. By now, you should have a deep understanding of how JSON parsing works and the ability to build your own parser for any grammar.

The parser we built handles all standard JSON, produces helpful error messages, supports the reviver parameter, and can be extended with non-standard features. While it is slower than native JSON.parse(), it is educational, customizable, and useful for specific use cases.

Key takeaways: parsing is about understanding grammar and systematically processing input; error handling is as important as successful parsing; testing is essential and should cover both valid and invalid inputs; and performance matters for large inputs but should not come at the cost of correctness.
