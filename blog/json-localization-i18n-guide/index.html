Using JSON for Localization: A Complete i18n Implementation Guide

Published: 2026-05-02 · 4-6 min read

Every major i18n library uses JSON as its translation format. JSON's hierarchical structure maps naturally to organized, namespaced translation keys. The flat alternative — each key as a plain string — loses structure quickly. JSON lets you group translations by feature, component, or page, making large translation files manageable. This guide covers patterns used in production applications serving millions of users across dozens of languages.

Basic Translation File Structure

{
  "app_name": "My Application",
  "nav": {
    "home": "Home",
    "about": "About",
    "contact": "Contact",
    "sign_in": "Sign In",
    "sign_out": "Sign Out"
  },
  "auth": {
    "login_title": "Welcome back",
    "email_label": "Email address",
    "password_label": "Password",
    "forgot_password": "Forgot your password?",
    "login_button": "Sign In",
    "no_account": "Don't have an account?",
    "register_link": "Create one"
  },
  "errors": {
    "required": "This field is required",
    "invalid_email": "Please enter a valid email",
    "server_error": "Something went wrong. Please try again.",
    "not_found": "The requested resource was not found"
  }
}

String Interpolation

{
  "user": {
    "welcome": "Welcome, {{name}}!",
    "last_login": "Last login: {{date}}",
    "subscription": "You have {{days}} days left on your {{plan}} plan",
    "profile_updated": "Profile updated at {{time}}"
  },
  "items": {
    "cart_count": "{{count}} items in your cart",
    "search_results": "Showing {{from}}-{{to}} of {{total}} results"
  }
}

// Usage with i18next:
// t('user.welcome', { name: 'Alice' })
// → "Welcome, Alice!"

// t('items.search_results', { from: 1, to: 20, total: 154 })
// → "Showing 1-20 of 154 results"

Pluralization

{
  "messages": {
    "inbox_zero": "No messages",
    "inbox_one": "1 message",
    "inbox_other": "{{count}} messages"
  },
  "time": {
    "minute_one": "1 minute",
    "minute_other": "{{count}} minutes",
    "hour_one": "1 hour",
    "hour_other": "{{count}} hours",
    "day_one": "Yesterday",
    "day_other": "{{count}} days ago"
  }
}

// i18next automatically picks the right form:
// t('messages.inbox', { count: 0 })  → "No messages"
// t('messages.inbox', { count: 1 })  → "1 message"
// t('messages.inbox', { count: 42 }) → "42 messages"

// Languages with complex plural rules (Arabic, Russian):
{
  "files_zero": "No files",
  "files_one": "{{count}} file",
  "files_two": "{{count}} files",
  "files_few": "{{count}} files",
  "files_many": "{{count}} files",
  "files_other": "{{count}} files"
}

File Organization at Scale

// Directory structure — namespace per feature/page:
// locales/
//   en/
//     common.json     ← shared across entire app
//     auth.json       ← login, register, forgot password
//     dashboard.json  ← dashboard page
//     settings.json   ← user settings
//     errors.json     ← error messages
//   fr/
//     common.json
//     auth.json
//     ... (same files, translated content)
//   zh/
//     ...

// common.json
{
  "save": "Save",
  "cancel": "Cancel",
  "delete": "Delete",
  "loading": "Loading...",
  "error": "Error",
  "success": "Success"
}

i18next Setup (React)

// i18n.js — configuration
import i18next from 'i18next';
import { initReactI18next } from 'react-i18next';

// Static import (small apps)
import enCommon from './locales/en/common.json';
import enAuth from './locales/en/auth.json';
import frCommon from './locales/fr/common.json';

i18next.use(initReactI18next).init({
  lng: 'en',
  fallbackLng: 'en',
  resources: {
    en: { common: enCommon, auth: enAuth },
    fr: { common: frCommon }
  },
  ns: ['common', 'auth'],
  defaultNS: 'common',
  interpolation: { escapeValue: false }
});

// Component usage
import { useTranslation } from 'react-i18next';

function LoginForm() {
  const { t } = useTranslation('auth');
  const { t: tCommon } = useTranslation('common');

  return (
    <form>
      <h1>{t('login.title')}</h1>
      <button type="submit">{t('login.submit')}</button>
      <button type="button" onClick={onCancel}>{tCommon('cancel')}</button>
    </form>
  );
}

Translation Management Tips

The biggest challenge with JSON i18n is keeping translation files synchronized across languages. When a developer adds a new key in English, the French and Japanese files immediately become incomplete. Tools like i18next-scanner extract all translation keys from source code and generate missing entries. CI pipelines can run these checks to prevent incomplete translations from reaching production. Consider using a translation management platform (Phrase, Lokalise, Crowdin) that reads your JSON format directly and provides a workflow for translators.

Key Takeaways

  • Use nested JSON objects to group translations by feature or page — avoid flat key lists at scale
  • Use {{variable}} interpolation syntax for dynamic content — keep HTML out of translation values
  • Handle pluralization with _zero, _one, _other suffix convention (i18next standard)
  • Organize large apps into namespaces (one JSON file per feature) — lazy-load namespaces not needed on first render
  • Automate key extraction with i18next-scanner to keep translation files in sync with source code
  • Never concatenate translated strings — use interpolation, or the word order will break for right-to-left and agglutinative languages
