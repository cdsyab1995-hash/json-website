JSON and TypeScript: Type-Safe JSON Parsing, Validation, and API Integration

Published: 2026-05-07 · 4-6 min read

TypeScript gives you compile-time type safety — but JSON comes from the network at runtime. JSON.parse() returns any. When you write const user = JSON.parse(response) as User, TypeScript trusts you completely, even if the API returns something completely different. This guide shows you how to close the gap between TypeScript's static types and the dynamic reality of JSON data, using runtime validation libraries that generate TypeScript types from your schemas.

The disconnect between TypeScript's static type system and runtime JSON data is one of the most common sources of bugs in TypeScript applications. Compile-time types disappear at runtime — TypeScript's type system exists only during development and compilation. The moment your code runs, there is no TypeScript, only JavaScript. JSON data arrives from external sources: REST APIs, WebSockets, localStorage, URL parameters, and database results. None of these sources have any idea what TypeScript types you've declared, and none of them enforce your interfaces. A backend developer might change a field name, a third-party API might add unexpected fields, or a legacy endpoint might return null where you expect a string. TypeScript won't tell you. Your runtime code will silently work with the wrong shape, producing bugs that are subtle, hard to reproduce, and far from the point of origin.

The solution is runtime validation — a library that checks the shape of data at the moment it arrives, before it enters your typed world. This guide covers the leading approaches, from the Zod library (the most popular choice in 2026) to generating types from existing JSON Schema and OpenAPI specifications. By the end, you'll have a systematic approach to handling JSON in TypeScript that is both type-safe at compile time and validated at runtime.

The Problem with Type Assertions

// This TypeScript compiles fine but silently crashes at runtime
interface User {
  id: number;
  name: string;
  email: string;
}

// DANGEROUS: No runtime validation
async function getUser(id: number): Promise<User> {
  const response = await fetch(`/api/users/${id}`);
  const data = await response.json();
  return data as User; // ← trusts the API blindly
}

// If the API returns { id: "abc", email: null, role: "admin" }
// TypeScript won't catch it — your code will fail later with
// confusing runtime errors far from the source

Runtime Validation with Zod

import { z } from 'zod';

// Define schema once — get TypeScript type automatically
const UserSchema = z.object({
  id: z.number().int().positive(),
  name: z.string().min(1),
  email: z.string().email(),
  role: z.enum(['admin', 'user', 'viewer']).default('user'),
  createdAt: z.string().datetime(),
  tags: z.array(z.string()).optional(),
});

// TypeScript type is inferred from the schema
type User = z.infer<typeof UserSchema>;
// Equivalent to: { id: number; name: string; email: string; ... }

// Safe parsing: never throws, returns success/error
async function getUser(id: number): Promise<User | null> {
  const response = await fetch(`/api/users/${id}`);
  const json = await response.json();
  
  const result = UserSchema.safeParse(json);
  
  if (!result.success) {
    console.error('API returned invalid user:', result.error.format());
    return null;
  }
  
  return result.data; // Fully typed User
}

JSON Schema to TypeScript Types

If you already have JSON Schema definitions (from OpenAPI specs, database schemas, or external APIs), you don't need to rewrite them in Zod. The json-schema-to-typescript package generates TypeScript interfaces directly from JSON Schema files. Run npx json-schema-to-typescript schema.json > types.ts to get typed interfaces. For OpenAPI specs, openapi-typescript generates complete, typed API client interfaces in seconds.

This approach is particularly valuable when you're working with a team that maintains a JSON Schema as the source of truth. Backend teams often publish their API schemas as JSON Schema or OpenAPI documents — instead of manually translating these to TypeScript interfaces, generate them automatically. The workflow is: backend publishes OpenAPI spec, frontend runs npx openapi-typescript openapi.yaml -o types.ts, and every field, endpoint, and response type is available as a TypeScript interface. When the backend changes the API, regenerating the types immediately surfaces every place in the frontend that needs updating.

The json-schema-to-typescript package handles complex schemas including nested objects, arrays of typed items, union types, enums, optional fields with defaults, and recursive schemas. It supports JSON Schema Draft 4, 6, and 7, making it compatible with schemas generated by virtually any tool in the ecosystem — Pydantic, Marshmallow, Ajv, and more. The generated TypeScript is clean, commented, and ready to use without modification in most cases.

Validating JSON at API Boundaries

The key principle of type-safe JSON in TypeScript is to validate at every trust boundary — points where untyped data enters your application. The most common boundaries are API responses, localStorage and sessionStorage reads, URL search parameters, environment variables, and file reads. Each of these can return data that doesn't match what you expect. A Zod schema at each boundary transforms unknown data into fully typed values, or catches the error before it propagates.

For API responses, always validate immediately after response.json() — before passing the data to any component or function. For storage reads, validate on retrieval, not on write — the data you stored six months ago might not match the schema you have today after a deployment. For URL parameters, use Zod's z.string().url(), z.coerce.number(), and z.enum() to parse and validate query string values. Treating these as unvalidated strings is a common source of subtle bugs when users bookmark URLs with old parameter formats.

Type-Safe JSON Storage

import { z } from 'zod';

// Type-safe localStorage wrapper
function createStorage<T>(key: string, schema: z.ZodType<T>) {
  return {
    get(): T | null {
      try {
        const raw = localStorage.getItem(key);
        if (!raw) return null;
        const parsed = JSON.parse(raw);
        return schema.parse(parsed); // Throws if invalid
      } catch {
        localStorage.removeItem(key); // Clear corrupted data
        return null;
      }
    },
    set(value: T): void {
      localStorage.setItem(key, JSON.stringify(value));
    }
  };
}

// Usage
const ThemeSchema = z.object({
  mode: z.enum(['light', 'dark']),
  fontSize: z.number().min(12).max(24),
});

const themeStorage = createStorage('theme', ThemeSchema);
const theme = themeStorage.get(); // Returns z.infer<typeof ThemeSchema> | null

This wrapper pattern handles three scenarios cleanly: the key doesn't exist (returns null), the stored JSON is corrupted or invalid (clears the key and returns null), and the data matches the schema (returns a fully typed value). The key insight is that schema.parse() will throw if the data doesn't match — and that exception is caught, the bad data is cleaned up, and the application continues with a sensible default rather than crashing.

Zod vs Other Validation Libraries

Zod is the most popular choice for TypeScript runtime validation in 2026, but it's not the only option. Valibot is a newer library that produces significantly smaller bundle sizes (under 1KB for basic schemas vs Zod's ~15KB) — a meaningful difference for client-side code. Yup is an older library with good ecosystem support and a slightly different API. Typebox generates both TypeScript types and JSON Schema simultaneously, making it ideal when you need to share schemas between frontend and backend. The choice depends on your constraints: Zod for full-featured TypeScript-first validation, Valibot for bundle-size-sensitive applications, and Typebox when JSON Schema interoperability matters.

Regardless of which library you choose, the principle is the same: define your schema once, infer the TypeScript type from it, and validate data at runtime before it enters your typed world. Never maintain a TypeScript interface and a Zod schema separately — they will drift. Let the schema be the source of truth, and derive the type from it.

Key Takeaways

  • Never use as Type to cast JSON.parse results — it silences real runtime errors
  • Use Zod's safeParse() for API responses — it never throws and handles errors explicitly
  • Infer TypeScript types from Zod schemas with z.infer<typeof Schema> to avoid drift between runtime and compile-time types
  • Generate types from existing JSON Schema or OpenAPI specs rather than rewriting from scratch — use json-schema-to-typescript or openapi-typescript
  • Validate JSON at every trust boundary: API responses, localStorage, URL parameters, and environment variables
  • Consider Valibot for bundle-size-sensitive apps; Typebox when you need JSON Schema and TypeScript types from one source
  • The schema is the source of truth — derive TypeScript types from it, never maintain them separately
