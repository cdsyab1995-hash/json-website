JSON vs YAML: When to Use Each Format in Modern Development

Published: 2026-05-01 · 4-6 min read

JSON and YAML serve overlapping but distinct roles in modern development. JSON is the undisputed king of data exchange — it's what APIs speak, what databases store, and what JavaScript parses natively. YAML excels at configuration files — Docker Compose, Kubernetes manifests, CI/CD pipelines, and application configs. Understanding when to use each saves time and prevents format-related bugs.

YAML Strengths: Configuration Files

# docker-compose.yml — YAML makes config readable
version: '3.8'
services:
  web:
    build: .
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=production
      - DATABASE_URL=postgres://db:5432/app
    depends_on:
      - db
      - redis

  db:
    image: postgres:15-alpine
    volumes:
      - pgdata:/var/lib/postgresql/data
    environment:
      - POSTGRES_PASSWORD=${DB_PASSWORD}  # env var substitution

  redis:
    image: redis:7-alpine

JSON Strengths: APIs and Data Exchange

JSON is the default format for REST APIs, GraphQL, WebSocket messages, and server-sent events. Every programming language has a built-in or first-party JSON parser. It's safe by default — no code execution, no file inclusion, no surprises. JSON is also more compact than YAML for the same data, which matters when you're transferring millions of API responses per day.

YAML Security Risks

# YAML allows type coercion that can be surprising
# This YAML:
user_input: "true"
admin: true
password: null
count: 123

# Parses to:
{ "user_input": true, "admin": true, "password": null, "count": 123 }

# Notice: "true" became a boolean! Strings that look like
# booleans, numbers, or null are auto-coerced.
# This can cause bugs when you expected strings.

# The worst case — unsafe YAML features in Ruby/Python:
# !!python/object/apply:os.system ['rm -rf /']
# This EXECUTES code during parsing. Never parse untrusted YAML
# with unsafe loaders.

# SAFE parsing in Python:
import yaml
with open('config.yml') as f:
    config = yaml.safe_load(f)  # safe_load prevents code execution

# NEVER use yaml.load() with untrusted input — it's like eval()

Converting Between Formats

# Python: Convert YAML to JSON
import json, yaml

with open('config.yml') as f:
    data = yaml.safe_load(f)

with open('config.json', 'w') as f:
    json.dump(data, f, indent=2)

# JavaScript: Convert JSON to YAML (using js-yaml)
import YAML from 'js-yaml';
import fs from 'fs';

const data = JSON.parse(fs.readFileSync('config.json', 'utf-8'));
const yamlStr = YAML.dump(data);
fs.writeFileSync('config.yml', yamlStr);

Key Takeaways

  • Use JSON for APIs, data exchange, and anything parsed by machines — it's safe, fast, and universal
  • Use YAML for configuration files written by humans — comments, readability, and multi-line strings matter
  • Never use YAML for untrusted input — use yaml.safe_load() in Python
  • YAML's type coercion (strings to booleans/numbers) causes subtle bugs — quote strings that look like types
  • Many tools accept both formats — choose based on your team's convention and the file's purpose
  • For configs parsed only by your own code, either format works — pick one and be consistent
