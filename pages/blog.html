🔍 JSONata: The XPath for JSON - Advanced Query and Transformation Language for US Developers

Published: April 17, 2026 | Reading time: 7 minutes | Target: US & Canada Developers

While JSONPath handles basic extraction, JSONata transforms JSON like a database query. Learn how this powerful language is revolutionizing data processing for US enterprise APIs and AI pipelines.

Why JSONata Matters in 2026

In 2026, APIs return increasingly complex nested structures. JSONPath gives you extraction—JSONata gives you transformation. Used by Telegraf, Node-RED, and major US cloud platforms, JSONata enables complex data reshaping without writing JavaScript loops.

// Input: Complex API response
{
  "orders": [
    {"id": "A001", "items": [{"sku": "X1", "qty": 2, "price": 29.99}], "customer": {"name": "John", "state": "CA"}},
    {"id": "A002", "items": [{"sku": "Y2", "qty": 1, "price": 49.99}], "customer": {"name": "Jane", "state": "NY"}}
  ]
}

// JSONata Query:
/orders.{"orderId": id, "customer": customer.name, "state": customer.state, "total": $sum(items.price * items.qty)}

// Result:
[
  {"orderId": "A001", "customer": "John", "state": "CA", "total": 59.98},
  {"orderId": "A002", "customer": "Jane", "state": "NY", "total": 49.99}
]

Core JSONata Syntax for US Developers

1. Path Expressions (Like XPath)

JSONata uses dot notation inspired by XPath's location paths:

product.data.details.price      // Single property
products[0].name                 // Array index
items[0 to 2]                    // Array slice (first 3)
data[*].price                    // Wildcard (*) for all items

2. Filtering with Predicates

orders[status = "shipped"]                    // Filter by value
products[price > 100 and stock > 0]           // Multiple conditions
data[sku ~> /^W\d+/]                          // Regex matching

3. Transformation & Restructuring

// Transform shape completely
items.{product: name, value: price * qty, taxable: price > 20}

// Recursive flatten with @
data.product.@{"sku": sku, "subtotal": price * quantity}

Built-in Functions for Production

JSONata includes 50+ functions for common operations:

// String functions
$uppercase(name)
$trim(description)
$split(csv_data, ",")
$contains(email, "@")

// Aggregation functions
$sum(order.items.price)
$average(scores)
$count(products)
$max(values)
$min(prices)

// Date/time handling
$now()                           // Current timestamp
$fromMillis(timestamp)           // Convert Unix ms to date
$formatDate($now(), "[Y0001]-[M01]-[D01]")  // "2026-04-17"

Real-World Use Cases for US Enterprises

1. ETL Pipeline Transformations

AWS Lambda + JSONata for data normalization:

// Transform Salesforce → internal format
{
  "crm_leads": salesforce_data.records.{
    "externalId": Id,
    "name": Name,
    "email": Email,
    "score": LeadScore,

2. IoT Data Processing (Telegraf)

Transform sensor data before storing:

{
  "deviceId": device,
  "timestamp": $fromMillis(ts),
  "readings": sensors.{sensor: name, value: value, unit: unit},
  "alerts": readings[value > threshold].{sensor: name, severity: "high"}
}

3. API Response Normalization

// Standardize multi-source API responses
$merge([
  response1.data,
  response2.items.{"key": id, "value": data}
]).{"id": key, "amount": value, "processed": $now()}

Performance Considerations

JSONata benchmarks show 3-5x faster execution vs manual JavaScript loops for transformation tasks. For production:

  • Pre-compile expressions in Node.js for repeated use
  • Use $lazy() for large arrays to avoid full scan
  • Cache compiled expressions in serverless functions

🛡️ Zod v4 + JSON Schema: Runtime Validation for AI Agent Responses in 2026

Published: April 16, 2026 | Reading time: 8 minutes | Target: US & Canada Developers

Your TypeScript types compile perfectly, your tests pass — and then your AI agent crashes in production because the LLM returned { "response": "yes" } instead of the expected { "answer": "yes", "confidence": 0.9 }. This is the silent killer of AI-powered applications. Zod v4, released August 2025, has become the standard defense layer for US developers building production AI agents.

1. Why TypeScript Types Alone Aren't Enough for LLM Responses

TypeScript type checking happens at compile time. LLM responses arrive at runtime. The moment you write JSON.parse(llmResponse) as MyType, you've added a type annotation with zero actual validation:

// ❌ TypeScript is lying to you here async function getAgentAction(prompt: string): Promise<AgentAction> { const raw = await llm.complete(prompt); return JSON.parse(raw) as AgentAction; // No actual runtime check! } // The LLM might return { "action": null, "params": "search" } // TypeScript won't catch this — Zod will

For every AI agent running in production, LLM providers can silently change response formats between model updates. Zod v4 turns "I hope this matches" into "I guarantee this matches — or I know exactly why it doesn't."

2. Zod v4: Key Features for AI Workflows

Released in August 2025, Zod v4 brings significant improvements specifically useful for AI workloads:

  • 14x faster string parsing — critical when validating thousands of LLM responses per minute
  • Built-in JSON Schema generation — .toJSONSchema() generates schemas compatible with OpenAI's Structured Outputs API
  • Zod Mini (1.9KB gzip) — tree-shakeable for edge agent deployments
  • Typed metadata — attach descriptions for agent self-introspection

3. Defining Agent Response Schemas

Here's how to define a robust schema for an AI agent that can search, calculate, or respond:

import { z } from 'zod/v4'; // Define the tool call schema const AgentActionSchema = z.discriminatedUnion('action', [ z.object({ action: z.literal('search'), params: z.object({ query: z.string().min(1).max(500), filters: z.array(z.string()).optional() }) }), z.object({ action: z.literal('calculate'), params: z.object({ expression: z.string().regex(/^[\d\s\+\-\*\/\(\)\.]+$/), precision: z.number().int().min(0).max(10).default(2) }) }), z.object({ action: z.literal('respond'), params: z.object({ message: z.string().min(1).max(2000), confidence: z.number().min(0).max(1) }) }) ]); type AgentAction = z.infer<typeof AgentActionSchema>;

4. The Retry Loop Pattern

The key to production-ready AI validation is feeding Zod errors back to the LLM for self-correction:

async function getValidatedAction( prompt: string, maxRetries = 3 ): Promise<AgentAction> { let lastError: string | null = null; for (let attempt = 0; attempt < maxRetries; attempt++) { const systemPrompt = lastError ? `Previous response was invalid: ${lastError}. Fix it.` : 'Respond with a valid JSON action.'; const raw = await llm.complete(prompt, { system: systemPrompt }); const result = AgentActionSchema.safeParse(JSON.parse(raw)); if (result.success) return result.data; // Format Zod errors for the LLM to understand lastError = result.error.issues .map(i => `${i.path.join('.')}: ${i.message}`) .join('; '); } throw new Error(`Agent failed after ${maxRetries} attempts`); }

5. Generating OpenAI-Compatible JSON Schemas

Zod v4's .toJSONSchema() integrates directly with OpenAI's Structured Outputs API, eliminating schema duplication:

import OpenAI from 'openai'; import { zodResponseFormat } from 'openai/helpers/zod'; import { z } from 'zod/v4'; const SummarySchema = z.object({ title: z.string().describe('Concise title under 10 words'), keyPoints: z.array(z.string()).min(3).max(5), sentiment: z.enum(['positive', 'neutral', 'negative']), confidence: z.number().min(0).max(1) }); const client = new OpenAI(); const response = await client.beta.chat.completions.parse({ model: 'gpt-4.2', messages: [{ role: 'user', content: 'Summarize this article...' }], response_format: zodResponseFormat(SummarySchema, 'summary') }); // Fully typed — no casting needed! const summary = response.choices[0].message.parsed;

  • Use z.discriminatedUnion() for multi-action agents — it provides faster parsing and clearer error messages than z.union()
  • Set additionalProperties: false equivalent with z.object().strict() to reject unexpected fields from LLMs
  • Track validation failure rates — target >99.5% success; drop below 98% means your schema or LLM prompt needs tuning
  • Remember: Zod validates structure, not truth — a confidence: 0.99 field can still be a hallucination

🎯 JSONPath Query Mastery: Advanced Extraction Patterns for US Production Systems

Published: April 16, 2026 | Reading time: 6 minutes | Target: US & Canada Developers

JSONPath is the XPath of JSON — but most developers only use the basics like $.name or $[0]. For US developers working with complex API responses, AI tool outputs, and multi-layer data structures, mastering advanced JSONPath patterns is essential. This guide covers production-ready techniques used by teams at Netflix, Airbnb, and Stripe.

1. The JSONPath Fundamentals (Quick Recap)

JSONPath expressions start with $ (root) and use dot or bracket notation:

  • $.store.book[0].title — Direct path to first book title
  • $.store.book[*].author — All book authors (wildcard)
  • $..price — Recursive descent to all prices

2. Filter Predicates: The Power Feature

Filter predicates use ?() syntax to select elements based on conditions:

// Extract expensive items over $100 $.store.products[?(@.price > 100)] // Multiple conditions with AND $.orders[?(@.status == 'shipped' && @.total > 500)] // Nested property access in filters $.users[?(@.address.city == 'Seattle')] // Array length comparison $.carts[?(@.items.length > 3)]

3. Array Slicing for Pagination

Python-style array slicing works perfectly for paginated data:

// First 10 items (offset:0, limit:10) $.apiResponse.data[:10] // Skip first 20, take next 10 (page 3) $.apiResponse.data[20:30] // Last 5 items (negative index) $.recentActivity[-5:] // Every other item (step) $.notifications[::2]

4. Function Expressions in JSONPath

JSONPath supports built-in functions for aggregation:

// Count items length($.store.book) // Sum values sum($.transactions[*].amount) // Get unique values distinct($.tags[*]) // Min/Max min($.prices[*]) max($.prices[*])

5. Real-World Production Patterns

Here are battle-tested patterns from US production systems:

// Extract all error messages from a log structure $..errors[*].message // Flatten nested API response for CSV export $.data[*].{name: name, value: metrics.value} // Deep search with recursive descent $..[?(@.id.startsWith('usr_'))] // Extract unique categories distinct($..category)

6. JavaScript Implementation

Using the popular jsonpath-plus library:

🤖 AI Tool Calling with JSON: How MCP is Standardizing Agent-Software Communication in 2026

Published: April 15, 2026 | Reading time: 7 minutes | Target: US & Canada Developers

In 2026, AI agents are no longer just generating text — they're executing actions. Claude can search your codebase, ChatGPT can query databases, and custom agents can deploy infrastructure. The bridge between an AI model and the real world? JSON utility calling.

1. How Tool Calling Works

Every major LLM provider — OpenAI, Anthropic, Google — now supports a tool calling interface where the model outputs structured JSON instead of natural language. The JSON contains the tool name and parameters:

{ "tool_use": { "id": "call_abc123", "name": "search_files", "input": { "path": "/src/components", "pattern": "*.tsx", "recursive": true } } }

Your application receives this JSON, executes the function, and returns the result back to the model as another JSON message. This loop continues until the task is complete.

2. MCP: The Emerging Standard

The Model Context Protocol (MCP), introduced by Anthropic and now backed by major players, is becoming the universal standard for AI tool interfaces. MCP defines:

  • Tool Schema: JSON Schema definitions for each tool's parameters
  • Message Types: Four standardized types — tools/list, tools/call, resources/read, prompts/get
  • Transport: Both stdio (local) and HTTP+SSE (remote) communication

3. Practical Example: Building a JSON-Powered Agent

Here's how to define a tool that an AI agent can call using the MCP format:

{ "name": "query_database", "description": "Execute a read-only SQL query", "inputSchema": { "type": "object", "properties": { "sql": { "type": "string", "description": "SQL SELECT query" }, "limit": { "type": "number", "default": 100 } }, "required": ["sql"] } }

The key insight: JSON Schema is the contract between human intent and AI execution. A well-designed schema with clear descriptions and proper constraints is what separates a reliable agent from one that hallucinates API calls.

4. Practices for US Developers

  • Always validate tool call inputs server-side — AI models can produce malformed JSON
  • Set timeouts on every tool execution to prevent hanging agent loops
  • Use MCP's tool listing to give agents a clear inventory of available actions
  • Return structured JSON results, not raw strings, for better agent comprehension

Tool calling with JSON is the foundation of the agent economy. Whether you're building internal automation or customer-facing AI products, understanding this pattern is now essential for every developer working in the US and Canada tech ecosystem.

🧩 JSON Merge Strategies: Deep vs Shallow vs Smart Override — A Practical Comparison

Published: April 13, 2026 | Reading time: 5 minutes

Merging two JSON objects sounds simple — but the devil is in the details. lodash.merge vs Object.assign vs recursive deep merge produce drastically different results. This guide compares three major merge strategies with real-world scenarios to help you choose the right approach.

1. The Three Strategies: What's the Difference?

Shallow Merge: Object.assign() only copies top-level properties. Nested objects remain shared references.

const a = { name: 'Alice', config: { theme: 'dark' } }; const b = { name: 'Bob', config: { lang: 'en' } }; const result = Object.assign({}, a, b); // Result: { name: 'Bob', config: { lang: 'en' } } // Warning: config.theme is lost!

Deep Merge: Recursively merges all nested properties.

import { merge } from 'lodash'; const result = merge({}, a, b); // Result: { name: 'Bob', config: { theme: 'dark', lang: 'en' } } // Both config objects are merged

Smart Merge: Deep merge behavior, but replaces arrays instead of merging array items.

2. JavaScript Implementation

function deepMerge(target, source) { const output = { ...target }; for (const key of Object.keys(source)) { if (source[key] && typeof source[key] === 'object' && !Array.isArray(source[key])) { output[key] = deepMerge(target[key] || {}, source[key]); } else { output[key] = source[key]; } } return output; }

3. When to Use Each

  • Override top-level config only: Use Object.assign()
  • Preserve all nested settings: Use deep merge
  • Merge user config with defaults: Smart merge recommended

⚡ Streaming JSON Parsing: Handling Real-Time AI Responses Efficiently

Published: April 11, 2026 | Reading time: 7 minutes

In 2026, streaming output has become a baseline expectation for AI applications. The typewriter effect in ChatGPT, the real-time reasoning scroll in Claude — these smooth experiences rely on a technology that trips up many developers: streaming JSON parsing. Unlike standard JSON responses, streaming responses deliver "in-progress" data that requires special handling to parse correctly.

1. What Is Streaming JSON? It's Not What You Think

When you call the streaming endpoint of OpenAI or Claude, the response looks roughly like this:

data: {"id":"chatcmpl-xxx","object":"chat.completion.chunk","choices":[{"delta":{"content":"Hello"}}]} data: {"id":"chatcmpl-xxx","object":"chat.completion.chunk","choices":[{"delta":{"content":" there"}}]} data: [DONE]

This is Server-Sent Events (SSE) format — each line starts with data: followed by an independent JSON object. Importantly: this is NOT one large JSON blob, but multiple independent JSON lines, known as NDJSON (Newline Delimited JSON).

2. Parsing Streaming Responses with Fetch + ReadableStream

Modern browsers can consume streaming responses using body.getReader():

const response = await fetch('/api/chat', { method: 'POST', body: JSON.stringify({ message: 'Hello' }) }); const reader = response.body.getReader(); const decoder = new TextDecoder(); let buffer = ''; while (true) { const { done, value } = await reader.read(); if (done) break; buffer += decoder.decode(value, { stream: true }); const lines = buffer.split('\n'); buffer = lines.pop(); // keep the incomplete last line for (const line of lines) { if (line.startsWith('data: ') && line !== 'data: [DONE]') { const json = JSON.parse(line.slice(6)); const content = json.choices[0]?.delta?.content || ''; process.stdout.write(content); // stream to UI } } }

3. Three Critical Edge Cases

  • Cross-chunk line breaks: Network chunks can split mid-line. Use a buffer to hold incomplete lines and wait for the next chunk to complete them before parsing.
  • Encoding issues: Always pass { stream: true } to TextDecoder so multi-byte characters (e.g., CJK) aren't incorrectly split across chunks.
  • [DONE] signal handling: OpenAI's format ends with data: [DONE]. Skip this line instead of trying to parse it as JSON.

4. Anthropic Claude Has a Slightly Different Format

Claude's streaming API uses explicit event type fields in each SSE message:

event: content_block_delta data: {"type":"content_block_delta","index":0,"delta":{"type":"text_delta","text":"Hello"}}

You need to check the event type first, then extract text from the corresponding data fields.

5. Pro Tip: Use JSON Tools to Debug Streaming Payloads

When debugging streaming JSON, try this workflow: copy individual data: lines into our JSON Formatter to inspect the structure, or use the JSON Compare tool to spot field differences between model versions quickly.

Summary

Key takeaways for streaming JSON: ① Process line-by-line, never parse the whole buffer at once; ② Use a buffer to handle cross-chunk lines; ③ Pass stream: true to TextDecoder; ④ Handle the [DONE] terminator correctly. Master these, and your AI app's UX will feel dramatically more responsive.

🔌 MCP Protocol Deep Dive: How JSON Powers the Model Context Revolution

Published: April 3, 2026 | Reading time: 7 minutes

Anthropic's Model Context Protocol (MCP) has quietly become one of the most important open standards in the AI ecosystem. As of early 2026, MCP is supported by Claude, Cursor, Zed, and dozens of third-party tools — and every single message in the protocol is a carefully structured JSON object.

What Is MCP?

MCP defines how AI assistants communicate with external tools, databases, and data sources. Think of it as a universal adapter: instead of every AI tool reinventing its own API, MCP provides a standardized JSON-based protocol that any client and server can speak.

The protocol is built on JSON-RPC 2.0, which means every request and response follows a predictable JSON structure:

{ "jsonrpc": "2.0", "id": 1, "method": "tools/call", "params": { "name": "search_files", "arguments": { "query": "JSON schema validation", "path": "/src" } } }

The Four Core MCP Message Types

Understanding these JSON structures is essential for building MCP-compatible tools:

  • Request: Client sends a method with params. Always has an id for correlation.
  • Response: Server returns a result or error matching the request id.
  • Notification: One-way message with no id, no response expected. Used for progress updates.
  • Tool Definition: JSON Schema describing what a tool accepts and returns, enabling the model to understand capabilities.

Tool Schemas: Where JSON Schema Shines

The most critical JSON structure in MCP is the tool definition. Each tool exposes an inputSchema that the model uses to understand what arguments to provide:

{ "name": "read_file", "description": "Read the contents of a file at the given path", "inputSchema": { "type": "object", "properties": { "path": { "type": "string", "description": "Absolute path to the file" } }, "required": ["path"], "additionalProperties": false } }

Notice the "additionalProperties": false — this strict schema ensures the model cannot hallucinate extra parameters that would break your tool implementation.

Validating MCP Payloads in Practice

When building MCP servers, always validate incoming JSON against your tool schemas before execution. A malformed tool call should return a structured JSON error, not a crash:

{ "jsonrpc": "2.0", "id": 1, "error": { "code": -32602, "message": "Invalid params: 'path' is required" } }

Why This Matters

MCP is standardizing how AI agents interact with the world — and it's all JSON under the hood. Mastering MCP's JSON structures today means you'll be ready to build the AI-native tools of tomorrow. The teams who understand the protocol deeply will ship more reliable AI integrations, faster.

What is JSON?

JSON (JavaScript Object Notation) is a lightweight data interchange format, popularized by Douglas Crockford in 2001. It is based on a subset of JavaScript but is language-independent, with nearly all modern programming languages providing native support for JSON.

Why Learn JSON?
  • API Development: Over 95% of web APIs use JSON for data transmission
  • Configuration Files: package.json, tsconfig.json, and more
  • Data Storage: NoSQL databases, configuration files
  • Logging Systems: Structured log recording

JSON Characteristics

  • Clean & Readable: More compact than XML with clear structure
  • Easy to Parse: Built-in support in all major programming languages
  • Cross-Platform: Language-independent, exchangeable between any systems
  • Type-Safe: Supports multiple data types

Common Errors & Solutions

Error 1: Unexpected token

Cause: Special characters in JSON string not escaped
Solution: Use escape characters like \", \\, \n 
// Incorrect {"text": "He said: "Hello""} // Correct {"text": "He said: \"Hello\""}

Error 2: Trailing comma

Cause: Comma after last element
Solution: Remove trailing comma 
// Incorrect [1, 2, 3, ] // Correct [1, 2, 3]

Error 3: Unquoted keys

Cause: JSON keys must be wrapped in double quotes
Solution: Add double quotes to all keys 
// Incorrect {name: "John"} // Correct {"name": "John"}

Error 4: Single quotes

Cause: JSON only supports double quotes
Solution: Replace single quotes with double quotes 
// Incorrect {'name': 'John'} // Correct {"name": "John"}
