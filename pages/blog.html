🔧 JSON Patch (RFC 6902): Stop Sending Full Objects — Use Partial Updates for Your APIs

Sending entire JSON payloads for small field updates wastes bandwidth and creates race conditions. JSON Patch (RFC 6902) provides a standardized way to describe exactly what changed.

2026-04-18 | 6-8 minutes

The Problem: PUT vs PATCH

Most REST APIs use PUT /users/123 with the entire user object when updating a single field. Want to change a display name? You send everything — email, address, preferences, phone number — back to the server. This approach has three major problems:

  • Bandwidth waste: Sending 50 fields to update 1 field is inefficient, especially on mobile networks
  • Race conditions: Two concurrent updates overwrite each other's changes — a classic lost-update problem
  • Over-exposure: Clients must know and send fields they may not even have permission to modify

Enter PATCH — but what format should the patch body use? That's where JSON Patch (RFC 6902) comes in.

How JSON Patch Works

A JSON Patch document is a JSON array of operation objects. Each object has an "op" field (the operation), a "path" (JSON Pointer targeting the location), and optionally a "value". Here's a real-world example:

[
  { "op": "replace", "path": "/displayName", "value": "Jane Doe" },
  { "op": "add", "path": "/preferences/notifications", "value": true },
  { "op": "remove", "path": "/legacyField" }
]

This patch tells the server: change the display name, enable notifications, and remove a deprecated field — all in one atomic request. The server applies operations sequentially, and if any operation fails, the entire patch is rolled back.

The Six Operations

  • add — Insert a value at the target path. If the target is an array, inserts before the specified index
  • remove — Delete the value at the target path
  • replace — Replace the value at the target path (equivalent to remove + add)
  • move — Move a value from one path to another
  • copy — Copy a value from one path to another
  • test — Verify that a value at the given path equals the specified value; fails the entire patch if not

The test Operation: Optimistic Concurrency Done Right

The test operation is JSON Patch's secret weapon for handling concurrent edits. Before applying changes, you can assert the current state:

[
  { "op": "test", "path": "/version", "value": 7 },
  { "op": "replace", "path": "/displayName", "value": "Jane Doe" },
  { "op": "replace", "path": "/version", "value": 8 }
]

If another client updated the resource and bumped the version to 8, the test fails and the server returns 409 Conflict. This is optimistic concurrency control without needing ETags or custom headers.

Real-World Adoption

Major platforms already use JSON Patch in production:

  • GitHub API: Uses JSON Patch for updating issue and PR labels in batch
  • Microsoft Graph API: Leverages JSON Patch for OneDrive and Outlook partial updates
  • OpenStack: Uses JSON Patch for resource attribute modification
  • Kubernetes: Strategic Merge Patch extends JSON Patch concepts for declarative config updates

JavaScript Implementation

Applying JSON Patch in JavaScript is straightforward using fast-json-patch, the most popular library with 3M+ weekly downloads:

import * as jsonpatch from 'fast-json-patch';

const user = { name: "John", email: "john@example.com", role: "user" };

const patch = [
  { op: "replace", path: "/name", value: "Jane" },
  { op: "add", path: "/preferences/theme", value: "dark" }
];

// Apply patch (mutates document)
jsonpatch.applyPatch(user, patch);

// Generate patch from diff
const observer = jsonpatch.observe(user);
user.email = "jane@example.com";
const changes = jsonpatch.generate(observer);
// [{ op: "replace", path: "/email", value: "jane@example.com" }]

When to Use JSON Patch

  • Collaborative editing: Real-time apps where multiple users edit the same resource
  • Mobile APIs: Reduce data transfer on metered connections
  • Offline-first apps: Queue patches locally and sync when online
  • Configuration management: Apply targeted changes to complex nested configs
  • CRUD optimization: Transform batch PUT operations into efficient partial updates

When NOT to Use JSON Patch

  • Your resources are small (under 10 fields)
  • Updates are infrequent and simple
  • Your team isn't familiar with JSON Pointer syntax
  • You need GraphQL-style fine-grained field selection (use GraphQL instead)

Key Takeaways

JSON Patch (RFC 6902) is an underused but powerful standard for partial API updates. Combined with the test operation, it provides atomic, conflict-free updates that scale. For any API handling collaborative editing, mobile clients, or large nested documents, JSON Patch deserves a place in your developer toolkit.
